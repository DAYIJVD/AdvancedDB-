##  ۱: عنوان فصل


**فصل ۱۰: داده‌های بزرگ (Big Data)**

### 📚 توضیح آموزشی

این فصل به بررسی چالش‌های ذخیره‌سازی، پردازش و تحلیل داده در مقیاس‌هایی می‌پردازد که سیستم‌های سنتی پایگاه داده از پس آن‌ها برنمی‌آیند. با رشد اینترنت، شبکه‌های اجتماعی و اینترنت اشیاء، روش‌های کاملاً جدیدی برای کار با داده ابداع شده‌اند که در این فصل بررسی می‌شوند.

---

## انگیزه

###  ۲: انگیزه



- حجم بسیار زیادی از داده در حال جمع‌آوری است
- ناشی از رشد وب، شبکه‌های اجتماعی و اخیراً اینترنت اشیاء
- گزارش‌های وب (Web logs) یکی از اولین منابع داده بودند
- تحلیل گزارش‌های وب ارزش بالایی برای تبلیغات، ساختاردهی سایت و پیشنهاد محتوا دارد
- **داده‌های بزرگ:** متمایز از داده‌های نسل قبلی پایگاه داده
  - **حجم (Volume):** مقادیر بسیار بیشتری از داده ذخیره می‌شود
  - **سرعت (Velocity):** نرخ درج داده بسیار بالاتر است
  - **تنوع (Variety):** انواع مختلفی از داده، فراتر از داده‌های رابطه‌ای

#### 📚 توضیح آموزشی

**مفهوم سه V در Big Data:**

| بُعد | توضیح | مثال |
|------|-------|------|
| Volume (حجم) | پتابایت‌ها تا اگزابایت‌ها | Facebook روزانه ۱۵+ پتابایت داده |
| Velocity (سرعت) | میلیون‌ها رویداد در ثانیه | Twitter: ۶۰۰۰ توییت در ثانیه |
| Variety (تنوع) | متن، تصویر، ویدیو، گراف، JSON | اینستاگرام: تصویر + متن + لایک + ... |

گاهی V چهارم و پنجم هم اضافه می‌شود:
- **Veracity (صحت):** چقدر می‌توان به داده اعتماد کرد؟
- **Value (ارزش):** چه اطلاعاتی واقعاً مفید است؟

این چالش‌ها باعث شدند معماری‌های کاملاً جدیدی ابداع شوند که در ادامه این فصل بررسی می‌شوند.

## پرس‌وجو از داده‌های بزرگ

###  ۳: پرس‌وجو از داده‌های بزرگ



- سیستم‌های پردازش تراکنش که به **مقیاس‌پذیری بسیار بالا** نیاز دارند
- بسیاری از برنامه‌ها حاضرند ویژگی‌های **ACID** و سایر امکانات پایگاه داده را فدا کنند تا به مقیاس‌پذیری بالا برسند
- سیستم‌های پردازش پرس‌وجو که نیاز دارند:
  - مقیاس‌پذیری بسیار بالا
  - پشتیبانی از داده‌های غیررابطه‌ای

#### 📚 توضیح آموزشی

**ACID در مقابل مقیاس‌پذیری — trade-off اساسی:**

خواص ACID (Atomicity, Consistency, Isolation, Durability) تضمین‌های قوی پایگاه‌های داده سنتی هستند. اما در مقیاس بزرگ، رعایت همه آن‌ها هزینه‌بر و کند است.

```
ACID کامل  ──────────────────────────────►  مقیاس‌پذیری نامحدود
    │                                                    │
 Oracle, PostgreSQL                        Cassandra, DynamoDB
 (تضمین قوی، مقیاس محدود)                (تضمین ضعیف‌تر، مقیاس بی‌نهایت)
```

سیستم‌های NoSQL مدل‌های ضعیف‌تری مثل **Eventual Consistency** را می‌پذیرند: داده‌ها در نهایت (نه بلافاصله) در همه نودها یکسان می‌شوند.

## سیستم‌های ذخیره‌سازی داده‌های بزرگ

###  ۴: سیستم‌های ذخیره‌سازی داده‌های بزرگ



- سیستم‌های فایل توزیع‌شده (Distributed File Systems)
- شاردینگ (Sharding) در پایگاه‌های داده چندگانه
- سیستم‌های ذخیره‌سازی کلید-مقدار (Key-Value Storage Systems)
- پایگاه‌های داده موازی و توزیع‌شده (Parallel and Distributed Databases)

#### 📚 توضیح آموزشی

**چهار رویکرد اصلی — از ساده به پیچیده:**

```
ساده‌تر
    │
    ├── ۱. شاردینگ: تقسیم DB های رابطه‌ای — مدیریت دستی
    │
    ├── ۲. سیستم فایل توزیع‌شده: برای فایل‌های بزرگ
    │
    ├── ۳. Key-Value Store: مقیاس‌پذیری خودکار، بدون SQL
    │
    └── ۴. پایگاه داده موازی: SQL روی چندین ماشین
    │
پیچیده‌تر
```

هر کدام برای سناریوهای خاصی مناسبند و انتخاب درست بستگی به نوع داده، الگوی دسترسی و نیازهای سیستم دارد.

---

###  ۵: سیستم‌های فایل توزیع‌شده



- یک سیستم فایل توزیع‌شده داده‌ها را روی مجموعه‌ای از ماشین‌ها ذخیره می‌کند اما **یک نمای واحد** از سیستم فایل ارائه می‌دهد
- مقیاس‌پذیری بالا برای برنامه‌های داده‌محور
  - مثال: ۱۰ هزار نود، ۱۰۰ میلیون فایل، ۱۰ پتابایت
- **ذخیره‌سازی افزونه** (Redundant) روی ماشین‌های ارزان‌قیمت و غیرقابل اعتماد
- فایل‌ها تکرار می‌شوند تا در برابر خرابی سخت‌افزار مقاوم باشند
- مثال‌ها: **Google File System (GFS)** و **Hadoop File System (HDFS)**

#### 📚 توضیح آموزشی

**ایده اصلی: Horizontal Scaling به جای Vertical Scaling**

```
Vertical Scaling (قدیمی):        Horizontal Scaling (مدرن):
یک سرور قوی‌تر بخر              هزاران سرور ارزان اضافه کن

     [سرور ۱۰۰K$]               [سرور$500] [سرور$500] [سرور$500]
          │                           │           │           │
     محدودیت فیزیکی                  └───────────┴───────────┘
                                           بی‌نهایت مقیاس‌پذیر
```

ویژگی‌های کلیدی:
- **Fault Tolerance:** اگر یک ماشین خراب شود، داده از روی نسخه‌های تکرار‌شده (معمولاً ۳ نسخه) قابل دسترس است
- **High Throughput:** داده‌ها به صورت موازی از چندین ماشین خوانده می‌شوند
- **Transparency:** کاربر یک سیستم فایل واحد می‌بیند، نه چندین ماشین

---

###  ۶: معماری سیستم فایل Hadoop



- یک فضای نام (Namespace) واحد برای کل کلاستر
- فایل‌ها به **بلوک‌هایی** تقسیم می‌شوند
  - اندازه معمول بلوک: **۶۴ مگابایت**
  - هر بلوک روی چندین DataNode تکرار می‌شود
- **کلاینت:**
  - موقعیت بلوک‌ها را از NameNode دریافت می‌کند
  - مستقیماً از DataNode به داده دسترسی پیدا می‌کند

#### 📚 توضیح آموزشی

**معماری دو نودی HDFS:**

```
                  ┌─────────────┐
                  │  NameNode   │  ← "کتابخانه‌دار"
                  │ (Master)    │
                  │ • نام فایل → لیست بلوک‌ها
                  │ • بلوک → DataNode ها
                  └──────┬──────┘
                         │ metadata
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
   ┌──────────┐   ┌──────────┐   ┌──────────┐
   │DataNode 1│   │DataNode 2│   │DataNode 3│
   │ Block A  │   │ Block A  │   │ Block B  │
   │ Block C  │   │ Block B  │   │ Block C  │
   └──────────┘   └──────────┘   └──────────┘
     (نسخه ۱)       (نسخه ۲)       (نسخه ۳)
```

- **NameNode** در RAM کار می‌کند — بسیار سریع اما Single Point of Failure
- **DataNode ها** هر ۳ ثانیه Heartbeat می‌فرستند؛ اگر NameNode جوابی نگرفت، آن نود را خراب می‌داند و replication آغاز می‌شود
- اندازه بلوک ۶۴MB انتخاب هوشمندانه: نه آنقدر کوچک که metadata زیاد شود، نه آنقدر بزرگ که streaming کند باشد

---

###  ۷: HDFS



- **NameNode:**
  - نام فایل را به لیست شناسه‌های بلوک نگاشت می‌کند
  - هر شناسه بلوک را به DataNodeهایی که نسخه‌ای از بلوک دارند نگاشت می‌کند
- **DataNode:** شناسه بلوک را به موقعیت فیزیکی روی دیسک نگاشت می‌کند
- **انسجام داده:**
  - مدل دسترسی **یک‌بار نوشتن، چندین‌بار خواندن** (Write-once-read-many)
  - کلاینت فقط می‌تواند به فایل‌های موجود **اضافه** کند
- سیستم‌های فایل توزیع‌شده برای **میلیون‌ها فایل بزرگ** مناسبند
- اما سربار بالا و کارایی ضعیف برای **میلیاردها تاپل کوچک**

#### 📚 توضیح آموزشی

**مدل Write-Once-Read-Many — چرا این محدودیت؟**

این طراحی عمدی است. در سیستم‌های Big Data معمولاً:
- داده‌ها یک‌بار نوشته می‌شوند (مثلاً لاگ‌ها)
- سپس بارها خوانده و تحلیل می‌شوند

مزایا:
- نیازی به lock mechanism پیچیده نیست
- بهینه‌سازی برای throughput بالای خواندن
- replication ساده‌تر می‌شود

**چه زمانی HDFS مناسب نیست؟**

| سناریو | مشکل |
|--------|------|
| فایل‌های کوچک زیاد | هر فایل metadata در NameNode دارد → از حافظه پر می‌شود |
| latency پایین | HDFS برای throughput بالا طراحی شده، نه latency پایین |
| به‌روزرسانی مکرر | Write-once مانع می‌شود |
| دسترسی تصادفی | برای Sequential Access بهینه است |

---

###  ۸: شاردینگ



- **شاردینگ:** تقسیم داده‌ها بین چندین پایگاه داده
- تقسیم‌بندی معمولاً بر اساس یک **کلید شارد** انجام می‌شود (مثل شناسه کاربر)
- برنامه باید ردیابی کند کدام رکوردها در کدام پایگاه داده هستند
- **مزایا:** مقیاس‌پذیری خوب، پیاده‌سازی آسان
- **معایب:**
  - شفاف نیست — برنامه باید مسیریابی پرس‌وجو را مدیریت کند
  - پرس‌وجوهای بین چند پایگاه داده دشوار است
  - جابجایی بار هنگام اضافه‌بار دشوار است
  - با افزایش تعداد پایگاه داده، احتمال خرابی بیشتر می‌شود

#### 📚 توضیح آموزشی

**انواع شاردینگ:**

```
Range-Based Sharding:
user_id 1-100K    → Shard A
user_id 100K-200K → Shard B
ساده است اما ممکن است یک شارد "داغ" (Hot Spot) شود

Hash-Based Sharding:
hash(user_id) % N → Shard Number
توزیع یکنواخت اما Range Query دشوار است

Directory-Based Sharding:
یک جدول راهنما: user_id 1234 → Shard C
انعطاف‌پذیر اما Single Point of Failure اضافه می‌کند
```

**مشکل Hot Spot:**
اگر کلید شارد تاریخ باشد و یک رویداد بزرگ (مثل جام جهانی) اتفاق بیفتد، تمام ترافیک به یک شارد می‌رود!

**Consistent Hashing — راه‌حل مدرن:**
وقتی شارد اضافه/حذف می‌کنی، فقط کمترین داده ممکن جابجا می‌شود (در مقابل hash modulo که همه چیز را redistribute می‌کند).

###  ۹: پایگاه‌های داده موازی و مخازن داده



**رویکرد ۱:** memcache یا سایر مکانیزم‌های کش — مقیاس‌پذیری محدود

**رویکرد ۲:** تقسیم‌بندی (شارد) داده‌ها در چندین سرور

**رویکرد ۳:** پایگاه‌های داده موازی — برای پشتیبانی تصمیم نه OLTP

**رویکرد ۴:** مخزن داده کلید-مقدار با پردازش موازی گسترده — تقسیم‌بندی، در دسترس بودن و ... کاملاً شفاف برای برنامه

#### 📚 توضیح آموزشی

**تفاوت OLTP و OLAP:**

| | OLTP | OLAP |
|--|------|------|
| هدف | تراکنش‌های کوچک و سریع | تحلیل‌های پیچیده |
| مثال | ثبت سفارش، خرید | گزارش فروش سالانه |
| رکوردهای درگیر | چند عدد | میلیون‌ها |
| نیاز | latency پایین، ACID | throughput بالا |
| پایگاه داده | PostgreSQL, MySQL | Greenplum, Vertica |

---

###  ۱۰: رویکرد ۱ — کش (Caching)



- **ایده:** قرار دادن یک کش درون‌حافظه سریع (مثل Memcached یا Redis) بین سرورهای برنامه و پایگاه داده اصلی
- **چرا محدود است:**
  - کش فقط برای خواندن داده‌های یکسان مفید است؛ نوشتن‌ها همچنان به پایگاه داده می‌رسند
  - اگر حجم کاری بزرگ‌تر از حافظه کش باشد، Cache Miss زیاد می‌شود
  - خود کش می‌تواند به گلوگاه تبدیل شود
  - **باطل‌سازی کش (Cache Invalidation)** دشوار است — داده‌های کهنه ممکن است نشان داده شود
- **نتیجه:** برای حجم کاری سنگین خواندن مناسب است، اما راه‌حل واقعی برای مقیاس‌پذیری نیست

#### 📚 توضیح آموزشی

**Redis در مقابل Memcached:**

| | Redis | Memcached |
|--|-------|-----------|
| ساختار داده | غنی (list, set, sorted set) | فقط key-value |
| Persistence | بله | خیر |
| Replication | بله | خیر |
| Threading | Single-threaded (mostly) | Multi-threaded |
| کاربرد | پیچیده‌تر، session store | ساده، cache محض |

**Cache Eviction Policies:**
- **LRU** (Least Recently Used): قدیمی‌ترین آیتم استفاده‌نشده حذف می‌شود
- **LFU** (Least Frequently Used): کم‌استفاده‌ترین آیتم حذف می‌شود
- **TTL** (Time to Live): بعد از X ثانیه خودکار پاک می‌شود

**مشکل Cache Invalidation:**
یکی از سخت‌ترین مسائل در CS است. روش‌ها: Cache-aside, Write-through, Write-behind — هر کدام trade-off خاص خود دارند.

---

###  ۱۱: رویکرد ۲ — شاردینگ



- **ایده:** تقسیم داده به شاردها؛ هر سرور زیرمجموعه‌ای از داده را دارد
- **Trade-off ها:**
  - به یک کلید شارد نیاز دارید
  - پرس‌وجوهایی که کلید شارد ندارند باید به همه شاردها ارسال شوند (Scatter-Gather)
  - عملیات بین چند شارد (join, transaction, aggregate) پیچیده یا غیرممکن می‌شوند
  - افزودن/حذف شارد نیاز به توازن‌بخشی مجدد داده دارد
- **نتیجه:** برای برنامه‌هایی که اکثر دسترسی‌ها به یک شارد محدود می‌شود مناسب است

#### 📚 توضیح آموزشی

**مثال واقعی — Instagram در ابتدا:**
Instagram از PostgreSQL با شاردینگ بر اساس user_id استفاده می‌کرد. هر query برای یک کاربر سریع بود، اما "trending posts" که به همه کاربران نیاز داشت بسیار کند بود.

**Shard Rebalancing — چالش اصلی:**
وقتی یک شارد پر می‌شود باید داده را به شارد جدید منتقل کنی در حالی که سیستم زنده است (Live Migration). این یکی از دشوارترین کارهاست.

MongoDB، Cassandra و DynamoDB این پیچیدگی‌ها را خودکار مدیریت می‌کنند.

---

###  ۱۲: رویکرد ۳ — پایگاه‌های داده موازی (تاریخی)



- **ایده:** استفاده از سیستم پایگاه داده طراحی‌شده برای اجرا روی ماشین‌های متعدد با تقسیم خودکار داده و موازی‌سازی پرس‌وجو
- **نکته تاریخی:** پایگاه‌های داده موازی اولیه (مثل Teradata, Greenplum) برای پشتیبانی تصمیم / انبار داده ساخته شدند
- OLTP نیاز به تأخیر کم و ACID دارد؛ OLAP بر throughput تمرکز دارد
- **نتیجه:** پایگاه‌های داده موازی برای تحلیل عالی هستند، اما OLTP به راه‌حل‌های تخصصی‌تر نیاز دارد

#### 📚 توضیح آموزشی

**تکامل پایگاه‌های داده موازی:**

```
دهه ۱۹۸۰: Teradata — اولین پایگاه داده موازی تجاری
دهه ۱۹۹۰: Parallel DB2, Parallel Oracle
دهه ۲۰۰۰: Greenplum, Vertica (ستون‌محور)
دهه ۲۰۱۰: Snowflake, Amazon Redshift (Cloud-native)
امروز:     BigQuery, Databricks (Lakehouse)
```

**پایگاه‌های داده ستون‌محور (Columnar):**
به جای ذخیره ردیف به ردیف، ستون به ستون ذخیره می‌کنند:
- وقتی می‌خواهی میانگین حقوق بگیری، فقط ستون حقوق می‌خوانی
- فشرده‌سازی بهتر (مقادیر یکسان در کنار هم)
- ۱۰ تا ۱۰۰ برابر سریع‌تر برای Analytical Queries

---

###  ۱۳: رویکرد ۴ — مخزن داده کلید-مقدار موازی گسترده



- **ایده:** ترکیب شاردینگ با تقسیم‌بندی شفاف، تکرار و در دسترس بودن بالا
- برنامه فقط بر اساس کلید می‌خواند یا می‌نویسد؛ سیستم خودکار درخواست را هدایت می‌کند
- **چه چیزی از دست می‌دهید:**
  - بدون join بین شاردها
  - بدون یکپارچگی کلید خارجی بین کلیدها
  - بدون تراکنش ACID
- **چرا پذیرفته می‌شود:** بسیاری از برنامه‌های بزرگ‌مقیاس به این ویژگی‌های رابطه‌ای نیاز ندارند

#### 📚 توضیح آموزشی

**مثال‌های واقعی:**

| سیستم | معماری | کاربرد اصلی |
|-------|--------|-------------|
| Amazon DynamoDB | Fully managed | session store, shopping cart |
| Apache Cassandra | Masterless | time-series, IoT |
| Google Bigtable | Wide-column | Google Search, Maps |
| MongoDB | Document | catalog, content |

**Denormalization — راه‌حل عدم join:**
وقتی join بین جداول ممکن نیست، داده‌ها را ادغام کن:
```
به جای: users + orders جدا

یک document واحد:
{
  "user_id": "123",
  "name": "Ali",
  "orders": [
    {"order_id": "456", "product": "کتاب", "price": 50000},
    {"order_id": "789", "product": "قلم", "price": 5000}
  ]
}
```
ذخیره‌سازی بیشتری می‌خواهد اما خواندن را بسیار سریع‌تر می‌کند.

###  ۱۸: پایگاه‌های داده موازی و توزیع‌شده



- پایگاه‌های داده موازی روی چندین ماشین (کلاستر) اجرا می‌شوند
- در دهه ۱۹۸۰ توسعه یافتند
- برای مقیاس کوچک‌تر (ده‌ها تا صدها ماشین) طراحی شده بودند
- مقیاس‌پذیری آسان فراهم نمی‌کردند
- معمولاً در صورت خرابی، **پرس‌وجو از ابتدا شروع می‌شود**
- **MapReduce می‌تواند اجرای پرس‌وجو را با دور زدن خرابی‌ها ادامه دهد**

#### 📚 توضیح آموزشی

**تفاوت Fault Recovery در سیستم‌های قدیمی و جدید:**

```
سیستم‌های قدیمی (Parallel DB):
نود خراب شد → transaction rollback → restart از ابتدا
در ۱۰ نود: معقول است
در ۱۰,۰۰۰ نود: احتمال خرابی همیشه بالاست → کار تمام نمی‌شود!

MapReduce / Spark:
نود خراب شد → فقط کار آن نود دوباره انجام می‌شود
بقیه کار از دست نمی‌رود → در مقیاس هزاران نود موفق است
```

این تفاوت بنیادین باعث شد MapReduce در مقیاس عظیم موفق شود.

###  ۱۹ و ۲۰: تکرار و سازگاری



- **دسترسی (Availability)** از طریق تکرار — حتی اگر یک نود خراب شده باشد، نسخه دیگری در دسترس است
- **سازگاری (Consistency)** برای داده‌های تکرارشده مهم است
  - اغلب با استفاده از **پروتکل‌های اکثریت** پیاده‌سازی می‌شود
  - مثلاً: ۳ نسخه، خواندن/نوشتن باید به ۲ نسخه دسترسی داشته باشند
- **پارتیشن شبکه:** در حضور پارتیشن، نمی‌توان هم دسترسی و هم سازگاری را تضمین کرد → **قضیه CAP**
- سیستم‌های بسیار بزرگ در یک نقطه پارتیشن می‌شوند → باید یکی را انتخاب کنی
  - پایگاه‌های داده سنتی **سازگاری** را انتخاب می‌کنند
  - اکثر برنامه‌های وب **دسترسی** را انتخاب می‌کنند

#### 📚 توضیح آموزشی

**قضیه CAP — مهم‌ترین مفهوم در سیستم‌های توزیع‌شده:**

```
         Consistency
              △
             /│\
            / │ \
           /  │  \
          /   │   \
         ▽────┴────▽
  Availability    Partition
                  Tolerance
```

- **C:** همه نودها در یک لحظه داده یکسانی می‌بینند
- **A:** هر درخواست جواب دریافت می‌کند (حتی اگر کامل‌ترین نباشد)
- **P:** سیستم حتی در صورت قطع شبکه کار می‌کند

قضیه CAP: **فقط می‌توانی دوتا از این سه را داشته باشی!**

در عمل P همیشه ضروری است، پس انتخاب واقعی: **CP یا AP**

| | CP | AP |
|--|----|----|
| رفتار در partition | درخواست را رد می‌کند | جواب می‌دهد (شاید قدیمی) |
| مثال | HBase, MongoDB | Cassandra, DynamoDB |
| مناسب برای | بانک، سیستم مالی | شبکه اجتماعی، لاگ |

**Majority Protocol:**
با ۳ نسخه: W=2 و R=2 → W+R > N=3 → همیشه حداقل یک نسخه مشترک بین خواندن و نوشتن = **Strong Consistency** تضمین می‌شود.

## پارادایم MapReduce

###  ۲۱: پارادایم MapReduce



- پلتفرمی برای **محاسبات موازی قابل اعتماد و مقیاس‌پذیر**
- مسائل محیط توزیع‌شده و موازی را از دید برنامه‌نویس **پنهان** می‌کند
- برنامه‌نویس منطق اصلی را از طریق توابع `map()` و `reduce()` فراهم می‌کند
- سیستم از موازی‌سازی محاسبات، هماهنگی و ... مراقبت می‌کند
- پیاده‌سازی‌های بسیار بزرگ‌مقیاس روی کلاسترهای با ۱۰۳ تا ۱۰۴ ماشین: **Google MapReduce، Hadoop**

#### 📚 توضیح آموزشی

**MapReduce از برنامه‌نویسی تابعی الهام گرفته:**

```python
# map: یک تابع را روی هر عنصر اعمال می‌کند
[1, 2, 3, 4] → map(x => x*2) → [2, 4, 6, 8]

# reduce: لیست را به یک مقدار کاهش می‌دهد
[2, 4, 6, 8] → reduce(sum) → 20
```

**نبوغ MapReduce:**
این عملیات طراحی شده‌اند تا به راحتی موازی شوند:
- `map` روی هزاران ماشین به صورت موازی اجرا می‌شود (هر ماشین بخشی از داده)
- `reduce` بعد از shuffle روی ماشین‌های مختلف اجرا می‌شود

مقاله اصلی گوگل (۲۰۰۴) نشان داد چگونه برنامه‌نویسان غیرمتخصص می‌توانند محاسبات بزرگ را روی هزاران ماشین اجرا کنند.

---

###  ۲۲: MapReduce — شمارش کلمات



**مسئله:** شمارش تعداد تکرار هر کلمه در مجموعه بزرگی از اسناد

**راه‌حل:**
1. تقسیم اسناد بین کارگران
2. هر کارگر سند را تجزیه کرده، تابع `map` خروجی `(word, count)` ارائه می‌دهد
3. جفت‌های `(word, count)` بر اساس کلمه بین کارگران تقسیم می‌شوند
4. برای هر کلمه در یک کارگر، تابع `reduce` تعدادها را جمع می‌کند

**ورودی:** `"One a penny, two a penny, hot cross buns."`

**خروجی map:** `("One",1), ("a",1), ("penny",1), ("two",1), ("a",1), ...`

**خروجی reduce:** `("One",1), ("a",2), ("penny",2), ("two",1), ...`

#### 📚 توضیح آموزشی

**مراحل تفصیلی Word Count:**

```
مرحله ۱ — Input Split:
فایل ۱۰۰GB به chunks تقسیم می‌شود
هر chunk به یک Map Worker داده می‌شود

مرحله ۲ — Map:
"سلام دنیا سلام" → (سلام,1), (دنیا,1), (سلام,1)

مرحله ۳ — Shuffle & Sort (کلیدی‌ترین بخش — نامرئی):
سیستم خودکار:
  • همه pairs را بر اساس key مرتب می‌کند
  • (سلام,[1,1]) , (دنیا,[1]) → به Reducer ها می‌فرستد

مرحله ۴ — Reduce:
(سلام,[1,1]) → (سلام, 2)
(دنیا,[1])   → (دنیا, 1)
```

**چرا مقیاس‌پذیر است؟**
- هر Map Task مستقل است → موازی روی ۱۰۰۰ ماشین
- هر Reduce Task مستقل است → موازی روی ۱۰۰۰ ماشین
- خرابی یک ماشین فقط آن بخش را دوباره اجرا می‌کند

###  ۲۳: شبه‌کد شمارش کلمات



```
map(String record):
  for each word in record:
    emit(word, 1)
  // اولین آرگومان emit کلید reduce نامیده می‌شود
  // group by روی کلید reduce انجام می‌شود

reduce(String key, List value_list):
  String word = key
  int count = 0
  for each value in value_list:
    count = count + value
  Output(word, count)
```

#### 📚 توضیح آموزشی

**تحلیل کد:**

تابع map:
- ورودی: یک رکورد (معمولاً یک خط متن)
- `emit()` به جای `return` چون می‌تواند چندین بار فراخوانی شود

تابع reduce:
- ورودی: یک کلید + **لیست تمام مقادیر** مرتبط با آن کلید
- این لیست به صورت streaming داده می‌شود — همه در حافظه نیستند!

**Combiner — بهینه‌سازی مهم:**
قبل از shuffle، یک mini-reduce محلی انجام بده:
```
روی یک ماشین: (a,1),(a,1),(a,1) → (a,3)
به جای ۳ پیام، فقط ۱ پیام به شبکه می‌رود
```
در Word Count می‌توانیم همان Reducer را به عنوان Combiner استفاده کنیم چون جمع کردن هم associative است و هم commutative.

###  ۲۴: مدل برنامه‌نویسی MapReduce



- الهام‌گرفته از عملیات map و reduce در زبان‌های برنامه‌نویسی تابعی مثل Lisp
- **ورودی:** مجموعه‌ای از جفت‌های کلید/مقدار
- کاربر دو تابع ارائه می‌دهد:
  - `map(k, v) → list(k1, v1)`
  - `reduce(k1, list(v1)) → v2`
- `(k1, v1)` یک جفت کلید/مقدار میانی است
- خروجی: مجموعه جفت‌های `(k1, v2)`

#### 📚 توضیح آموزشی

**مدل انتزاعی MapReduce:**

```
ورودی:     (k_in,  v_in)  pairs
              ↓ map
میانی:     (k_mid, v_mid) pairs — ممکن است چندین تا باشد
              ↓ shuffle & sort
گروه‌بندی: k_mid → [v_mid_1, v_mid_2, ...]
              ↓ reduce
خروجی:     (k_mid, v_out) pairs
```

**انعطاف‌پذیری مدل — چه چیزی می‌توانی با MapReduce بیان کنی؟**

| مسئله | Map | Reduce |
|-------|-----|--------|
| Word Count | (word, 1) | جمع |
| Grep | (خط شامل pattern, خط) | identity |
| Inverted Index | (word, docID) | لیست docID ها |
| PageRank | (node, rank/links) | جمع rank های ورودی |
| Join | (key, (table, row)) | merge کردن |

###  ۲۵: MapReduce مثال ۲ — پردازش لاگ



**فرمت لاگ:**
```
2013/02/21 10:31:22.00EST /slide-dir/11.ppt
2013/02/21 10:43:12.00EST /slide-dir/12.ppt
```

**هدف:** تعداد دفعاتی که هر فایل در پوشه slide-dir بین ۲۰۱۳/۰۱/۰۱ تا ۲۰۱۳/۰۱/۳۱ دسترسی داشته

**گزینه‌ها:**
- برنامه ترتیبی: خیلی کند برای داده‌های عظیم
- بارگذاری در پایگاه داده: گران، عملیات مستقیم روی فایل‌های لاگ ارزان‌تر است
- **پارادایم MapReduce ✓**

#### 📚 توضیح آموزشی

**چرا Log Processing مهم است؟**

- Netflix: کاربران چه محتوایی را می‌بینند؟ کجا رها می‌کنند؟
- Google: کدام صفحات بیشترین بازدید دارند؟
- Amazon: کاربران چه مسیری تا خرید دارند؟

**ابعاد مسئله:**
یک وب‌سایت بزرگ ممکن است روزانه ۱۰۰TB لاگ تولید کند. پردازش با یک ماشین: هفته‌ها. با MapReduce روی ۱۰۰۰ ماشین: چند ساعت!

**بهینه‌سازی کلیدی:**
فیلترینگ در `map` باعث می‌شود داده‌های غیرمرتبط اصلاً به مرحله shuffle نرسند و traffic شبکه به شدت کاهش یابد.

###  ۲۶: MapReduce — شمارش دسترسی به فایل‌ها



```java
map(String key, String record) {
  String date = attribute[0];
  String time = attribute[1];
  String filename = attribute[2];
  if (date between 2013/01/01 and 2013/01/31
      and filename starts with "/slide-dir/")
    emit(filename, 1);
}

reduce(String key, List recordlist) {
  String filename = key;
  int count = 0;
  for each record in recordlist
    count = count + 1;
  output(filename, count)
}
```

#### 📚 توضیح آموزشی

**تحلیل کد:**

map انجام می‌دهد:
1. هر خط لاگ را به ۳ قسمت تقسیم می‌کند
2. **فیلتر می‌کند:** فقط بازه زمانی درست و پوشه slide-dir
3. `emit(filename, 1)` می‌کند

reduce انجام می‌دهد:
1. کلید = نام فایل
2. تمام ۱ها را جمع می‌کند

**با Combiner:**
اگر روی یک ماشین ۱۰۰۰ بار به `12.ppt` دسترسی بود:
- بدون Combiner: ۱۰۰۰ پیام `(12.ppt, 1)` به شبکه
- با Combiner: ۱ پیام `(12.ppt, 1000)` به شبکه
- صرفه‌جویی ۹۹.۹٪ در ترافیک شبکه!

###  ۲۷: جریان شماتیک کلیدها و مقادیر



```
ورودی‌های map:    (mk1,mv1), (mk2,mv2), ..., (mkn,mvn)
خروجی‌های map:   (rk1,rv1), (rk7,rv2), (rk3,rv3), ...
پس از گروه‌بندی: rk1→[rv1,rv7,...], rk2→[rv8,...], ...
```

#### 📚 توضیح آموزشی

**Shuffle & Sort — گران‌ترین مرحله:**

این مرحله نامرئی است اما کلیدی‌ترین بخش MapReduce است:
1. خروجی‌های map را مرتب می‌کند
2. همه مقادیر با کلید یکسان را در کنار هم قرار می‌دهد
3. هر گروه را به یک Reducer می‌فرستد

این مرحله به شدت I/O-bound است و اغلب گلوگاه اصلی است.

**Data Skew:**
اگر یک key خیلی بیشتر از بقیه تکرار شود (مثل کلمه "the" در Word Count)، آن Reducer بسیار کند می‌شود. راه‌حل: hot key را در چندین Reducer تقسیم کن.

---

###  ۲۸: پردازش موازی کار MapReduce



```
برنامه کاربر → Master → Map Workers (خواندن و نوشتن محلی)
                      → Reduce Workers (خواندن از راه دور، مرتب‌سازی، نوشتن خروجی)
پارتیشن‌های ورودی → فایل‌های میانی → فایل‌های خروجی
```

#### 📚 توضیح آموزشی

**نقش Master:**
- تخصیص Map Tasks به ماشین‌های آزاد
- ردیابی وضعیت هر task (idle, in-progress, completed)
- **Speculative Execution:** اگر task در مدت معقولی تمام نشد، همان task را روی ماشین دیگری هم شروع کن

**Data Locality — بهینه‌سازی کلیدی:**
سعی می‌شود Map Task روی همان ماشینی اجرا شود که داده را دارد:
```
بدون Data Locality: داده از HDFS → شبکه → ماشین محاسباتی
با Data Locality:   Map Task روی همان ماشین DataNode اجرا می‌شود
```
این انتقال داده از شبکه را حذف می‌کند — صرفه‌جویی عظیم!

###  ۳۰–۳۴: کد Java در Hadoop



**انواع در Hadoop:**
`Mapper<LongWritable, Text, Text, IntWritable>` — ورودی کلید، ورودی مقدار، خروجی کلید، خروجی مقدار

**تابع Map:**
```java
public static class Map extends Mapper<LongWritable, Text, Text, IntWritable> {
  private final static IntWritable one = new IntWritable(1);
  private Text word = new Text();
  public void map(LongWritable key, Text value, Context context) {
    String line = value.toString();
    StringTokenizer tokenizer = new StringTokenizer(line);
    while (tokenizer.hasMoreTokens()) {
      word.set(tokenizer.nextToken());
      context.write(word, one);
    }
  }
}
```

**تابع Reduce:**
```java
public static class Reduce extends Reducer<Text, IntWritable, Text, IntWritable> {
  public void reduce(Text key, Iterable<IntWritable> values, Context context) {
    int sum = 0;
    for (IntWritable val : values) { sum += val.get(); }
    context.write(key, new IntWritable(sum));
  }
}
```

**برنامه کلی:**
```java
public class WordCount {
  public static void main(String[] args) throws Exception {
    Configuration conf = new Configuration();
    Job job = new Job(conf, "wordcount");
    job.setOutputKeyClass(Text.class);
    job.setOutputValueClass(IntWritable.class);
    job.setMapperClass(Map.class);
    job.setReducerClass(Reduce.class);
    job.setInputFormatClass(TextInputFormat.class);
    job.setOutputFormatClass(TextOutputFormat.class);
    FileInputFormat.addInputPath(job, new Path(args[0]));
    FileOutputFormat.setOutputPath(job, new Path(args[1]));
    job.waitForCompletion(true);
  }
}
```

#### 📚 توضیح آموزشی

**چرا Hadoop از انواع سفارشی استفاده می‌کند؟**

انواع اصلی Java (int, long, String) برای serialize/deserialize شبکه‌ای بهینه نیستند.
`IntWritable`, `LongWritable`, `Text` رابط `Writable` را پیاده‌سازی می‌کنند:
- `write(DataOutput out)`: سریالایز به bytes
- `readFields(DataInput in)`: دسریالایز از bytes

**بهینه‌سازی مهم در Map:**
```java
private final static IntWritable one = new IntWritable(1); // یک بار ساخته می‌شود
private Text word = new Text();                             // reuse می‌شود
```
اگر هر بار `new` صدا بزنی، Garbage Collection به شدت performance را کاهش می‌دهد.

**`Iterable<IntWritable>` در Reduce:**
این iterator streaming است — همه مقادیر در حافظه نیستند! اگر یک کلمه میلیون‌ها بار تکرار شده باشد، همه در RAM نمی‌آیند.

###  ۳۵: تجمیع محلی (Combiners)



- **Combiner ها:** تجمیع جزئی برای کاهش ترافیک شبکه
  - درون ماشین
  - و/یا در سطح rack
- در Hadoop، تابع reduce به طور پیش‌فرض توسط combiner استفاده می‌شود

#### 📚 توضیح آموزشی

**تأثیر Combiner بر ترافیک:**

```
بدون Combiner:
Map Machine 1: (the,1),(the,1),(the,1),...×100,000 → شبکه → Reducer

با Combiner:
Map Machine 1: (the,1),(the,1),(the,1),...×100,000
                      ↓ Combiner
                   (the, 100000) → شبکه → Reducer

کاهش ۱۰۰,۰۰۰× در ترافیک شبکه!
```

**شرط استفاده از Reducer به عنوان Combiner:**
عملیات باید هم **associative** باشد: `(a op b) op c = a op (b op c)`
هم **commutative** باشد: `a op b = b op c`

Combiner برای میانگین کار نمی‌کند:
```
میانگین [1,2,3] و [4,5] ≠ میانگین(میانگین[1,2,3], میانگین[4,5])
2                           ≠ میانگین(2, 4.5) = 3.25
```
راه‌حل: Combiner مجموع و تعداد را emit کند؛ Reducer تقسیم کند.

###  ۳۶ و ۳۷: MapReduce در مقابل پایگاه‌های داده



- MapReduce به طور گسترده استفاده می‌شود: PageRank، ایندکس کلمات کلیدی، تحلیل کلیک
- اجازه **کد رویه‌ای** در توابع map و reduce می‌دهد
- داده از **هر نوعی** را پردازش می‌کند
- بسیاری از کاربردها با SQL قابل بیان نیستند
- اما بسیاری از محاسبات با SQL **خیلی آسان‌تر** بیان می‌شوند
- عملیات رابطه‌ای (select, project, join, aggregation) می‌توانند با MapReduce بیان شوند
- **Apache Hive SQL، Apache Pig Latin** — SQL را به MapReduce ترجمه می‌کنند

#### 📚 توضیح آموزشی

**زمانی MapReduce بهتر است:**
- کد سفارشی و الگوریتم‌های غیرمعمول
- داده‌های نامنظم (HTML، لاگ‌های خام)
- Pipeline های پیچیده چند مرحله‌ای

**زمانی SQL بهتر است:**
- Aggregation ساده (COUNT, SUM, AVG)
- Join بین جداول ساختارمند
- کاربران غیر برنامه‌نویس (Data Analysts)

**تکامل SQL-on-Hadoop:**
```
نسل ۱: Hive → SQL به MapReduce ترجمه می‌شود (خیلی کند)
نسل ۲: Tez, Impala, Spark SQL → ۱۰-۱۰۰× سریع‌تر
نسل ۳: Presto/Trino, Databricks SQL → بهینه‌ترین
```

## فراتر از MapReduce

###  ۳۹: عملیات جبری



- موتورهای اجرای نسل فعلی:
  - به صورت بومی از عملیات جبری مثل join و aggregation پشتیبانی می‌کنند
  - به کاربران اجازه می‌دهند عملگرهای جبری خود را ایجاد کنند
  - از درخت‌های عملگرهای جبری که روی چندین نود به صورت موازی اجرا می‌شوند پشتیبانی می‌کنند
  - مثلاً **Apache Tez** و **Spark**
- Tez یک API سطح پایین ارائه می‌دهد؛ Hive روی Tez پرس‌وجوهای SQL را کامپایل می‌کند
- Spark یک API کاربرپسندتر ارائه می‌دهد

#### 📚 توضیح آموزشی

**چرا MapReduce کافی نبود؟**

محدودیت‌های MapReduce:
1. فقط دو مرحله: map → shuffle → reduce
2. خروجی هر مرحله به HDFS نوشته می‌شود → I/O گران
3. برای ۵۰ iteration: ۱۰۰ عملیات I/O دیسک!

**DAG در سیستم‌های جدید:**
```
MapReduce:  Map → Shuffle → Reduce → HDFS → Map → ...
              (I/O دیسک بین هر مرحله)

Spark/Tez:  Filter → Join → GroupBy → Aggregate
              (پیپلاین در حافظه، بدون I/O اضافه)
```

Query Optimizer می‌تواند ترتیب عملیات را بهینه کند — مثل filter را "push down" کند تا داده کمتری منتقل شود.

###  ۴۰: عملیات جبری در Spark



- **انتزاع RDD (Resilient Distributed Dataset):**
  - مجموعه‌ای از رکوردها که می‌توانند روی چندین ماشین ذخیره شوند
  - RDD ها می‌توانند با اعمال عملیات جبری روی سایر RDD ها ایجاد شوند
  - RDD ها می‌توانند به صورت **lazy** محاسبه شوند
- Spark از **Lambda expressions** جاوا ۸ استفاده می‌کند

#### 📚 توضیح آموزشی

**RDD — سه ویژگی اصلی:**

```
R = Resilient: اگر بخشی از دست رفت، با lineage graph دوباره محاسبه کن
D = Distributed: در RAM چندین ماشین ذخیره می‌شود
D = Dataset: مجموعه‌ای از رکوردها
```

**Lazy Evaluation — چرا مهم است؟**

```python
# این کد هیچ کاری نمی‌کند تا زمانی که action صدا نزنی!
result = data.filter(age > 18).filter(city == "Tehran").count()
#                                                        ↑ این action است
```

Spark می‌داند هر دو filter را می‌خواهی → آن‌ها را با هم ترکیب می‌کند → یک pass روی داده به جای دو pass.

**Transformations vs Actions:**
```
Transformations (تنبل): filter, map, flatMap, join → RDD جدید می‌سازند
Actions (بلادرنگ):       collect, count, saveAsTextFile → اجرا را trigger می‌کنند
```

###  ۴۱: شمارش کلمات در Spark



```java
JavaRDD<String> lines = sc.textFile(inputFile);
JavaRDD<String> words = lines.flatMap(
  s -> Arrays.asList(s.split(" ")).iterator()
);
JavaPairRDD<String, Integer> counts = words
  .mapToPair(s -> new Tuple2(s, 1))
  .reduceByKey((i1, i2) -> i1 + i2);
counts.saveAsTextFile(outputFile);
```

#### 📚 توضیح آموزشی

**مقایسه Hadoop و Spark برای Word Count:**

| | Hadoop | Spark |
|--|--------|-------|
| خطوط کد | ~۶۰ خط Java | ~۵ خط Java |
| سرعت | baseline | ۱۰-۱۰۰× سریع‌تر |
| I/O | دیسک بین مراحل | RAM |

**همان کد در Python (PySpark):**
```python
counts = (sc.textFile(inputFile)
  .flatMap(lambda line: line.split(" "))
  .map(lambda word: (word, 1))
  .reduceByKey(lambda a, b: a + b))
counts.saveAsTextFile(outputFile)
```

**چرا Spark سریع‌تر از Hadoop است؟**
1. **In-memory:** RDD ها در RAM نگه داشته می‌شوند نه HDFS
2. **No unnecessary I/O:** بین مراحل داده به دیسک نوشته نمی‌شود
3. **Pipelining:** عملیات بدون تبدیل به MapReduce
4. **Catalyst Optimizer:** بهینه‌سازی خودکار query ها

###  ۴۵: داده‌های جریانی و کاربردها



- **داده‌های جریانی:** داده‌هایی که به صورت پیوسته می‌رسند؛ در تضاد با data-at-rest
- **کاربردها:**
  - بازار سهام: جریانی از معاملات
  - سایت‌های تجارت الکترونیک: خریدها و جستجوها

#### 📚 توضیح آموزشی

**Batch vs Stream Processing:**

```
Batch (داده در سکون):
```

- Real-time Bidding در تبلیغات: تصمیم در ۱۰۰ میلی‌ثانیه!
- IoT: میلیون‌ها حسگر → تشخیص خرابی در لحظه

###  ۴۶ و ۴۷: پرس‌وجو از داده‌های جریانی



**رویکردها:**
- **Windowing:** تقسیم جریان به پنجره‌ها و اجرای پرس‌وجو روی پنجره‌ها (زمانی یا بر اساس تاپل)
- **Continuous Queries:** پرس‌وجوهای SQL که نتایج جزئی بر اساس جریان دیده‌شده خروجی می‌دهند
- **عملیات جبری روی جریان‌ها:** هر عملگر تاپل‌هایی از جریان مصرف می‌کند و تاپل‌هایی خروجی می‌دهد؛ می‌تواند وضعیت نگه دارد
- **تطبیق الگو (Pattern Matching):** سیستم‌های **CEP** (Complex Event Processing)

#### 📚 توضیح آموزشی

**Out-of-Order Events — چالش اصلی:**

در دنیای واقعی رویدادها ممکن است دیر برسند:
```
Event Time:      زمانی که رویداد واقعاً اتفاق افتاد
Processing Time: زمانی که سیستم آن را دید
Ingestion Time:  زمان ورود به سیستم streaming
```

**Watermarks — راه‌حل:**
"انتظار دارم رویدادهایی که ۵ دقیقه دیر باشند حداکثر ۱۰ دقیقه بعد برسند."
بعد از watermark، پنجره بسته می‌شود و نتیجه منتشر می‌شود.

**CEP مثال — تشخیص Brute Force:**
```java
// Flink CEP
Pattern<LoginEvent, ?> pattern = Pattern.<LoginEvent>begin("first")
  .where(event -> event.isFailed())
  .times(5)
  .within(Time.minutes(1))
  .next("success")
  .where(event -> event.isSuccess());
```

###  ۴۸: معماری‌های پردازش جریان



- بسیاری از سیستم‌های پردازش جریان کاملاً درون‌حافظه‌ای هستند
- **معماری Lambda:** جریان را به دو قسمت تقسیم کن
  - یک خروجی به سیستم پردازش جریان می‌رود
  - دیگری به پایگاه داده برای ذخیره‌سازی
  - پیاده‌سازی آسان و به طور گسترده استفاده می‌شود
  - اما اغلب منجر به **تکرار تلاش پرس‌وجو** می‌شود

#### 📚 توضیح آموزشی

**معماری Lambda:**

```
Data Source ──┬──► Batch Layer (Hadoop/Spark)  ──► Serving Layer ──► Query
              │                                     ↑
              └──► Speed Layer (Kafka+Flink)  ───────┘
```

- **Batch Layer:** دقیق اما با تأخیر (ساعت‌ها)
- **Speed Layer:** سریع اما تقریبی (میلی‌ثانیه)
- **Serving Layer:** ترکیب هر دو برای پاسخ به query

**مشکل Lambda:**
باید منطق پردازش را **دو بار** پیاده‌سازی کنی (batch و streaming). bug در یکی → inconsistency.

**معماری Kappa (بهتر):**
فقط streaming داری. برای reprocessing، داده‌های تاریخی را از Kafka replay کن.

###  ۴۹–۵۳: انواع پنجره در SQL Streaming



**Tumbling Window:** اندازه ثابت، بدون همپوشانی، تراز با زمان (مثلاً ساعتی)

**Hopping Window:** اندازه ثابت، همپوشانی دارد — تعریف: طول پنجره + فاصله hop
- مثال: پنجره ۱ ساعته که هر ۲۰ دقیقه محاسبه مجدد می‌شود

**Sliding Window:** مداوم در حال حرکت، رویداد-محور — با هر رویداد جدید یک پنجره جدید ایجاد می‌شود

**Session Window:** مبتنی بر فعالیت کاربر — شکاف‌ها مرزها را تعریف می‌کنند؛ بعد از timeout بی‌فعالی، پنجره بسته می‌شود

#### 📚 توضیح آموزشی

**مقایسه چهار نوع پنجره:**

```
Tumbling (ساده‌ترین):
[──1h──][──1h──][──1h──]
هر رویداد در دقیقاً یک پنجره

Hopping (همپوشانی):
[──1h──────]
     [──1h──────]
          [──1h──────]
(hop=20min) هر رویداد در چند پنجره

Sliding (دقیق‌ترین):
هر رویداد یک پنجره جدید ایجاد می‌کند
بیشترین محاسبات

Session (واقعی‌ترین):
[──user1 active──][timeout][──user1 active──]
[──user2 active──────────────]
هر session طول متفاوتی دارد
```

**کاربرد هر نوع:**
- Tumbling: گزارش‌های ساعتی/روزانه
- Hopping: میانگین متحرک، smoothing
- Sliding: anomaly detection در real-time
- Session: تحلیل رفتار کاربر، funnel analysis

###  ۵۵–۵۷: Stream Joins



**Stream-Relation Join:**
- اغلب نیاز دارید جریان را با یک جدول مرجع ثابت غنی کنید
- مثال: `orders(itemId, amount)` JOIN با `products(itemId, category)`
- یک طرف بی‌نهایت (جریان)، طرف دیگر محدود (جدول)
- نیازی به محدوده زمانی نیست

**Stream-Stream Join:**
- نمی‌توانید هر رویداد جریان A را با هر رویداد جریان B join کنید
- باید **محدود کنید** چقدر رویدادهای منطبق می‌توانند فاصله زمانی داشته باشند
- مثال: Ad Click JOIN با Conversion — فقط اگر در ۳۰ دقیقه بعد از click باشد
- بدون محدوده gap: باید همه تاریخ را در حافظه نگه داری!

#### 📚 توضیح آموزشی

**State Expiration — چرا Gap Bound ضروری است:**

```
بدون Gap Bound:
Stream A event (t=0) ──── ممکن است با Stream B event (t=∞) match شود
→ باید تمام تاریخ را در حافظه نگه داری → غیرممکن!

با Gap Bound (مثلاً 30 دقیقه):
هر event که ۳۰ دقیقه از آن گذشت و match نشد → از حافظه پاک می‌شود
→ State در حافظه bounded می‌شود → ممکن!
```

**Attribution Analysis — کاربرد واقعی:**
"کدام تبلیغ باعث این خرید شد؟"
- Last-click: آخرین click
- Multi-touch: همه clicks با وزن
- Time-decay: click های نزدیک‌تر وزن بیشتری دارند

###  ۵۸ و ۵۹: سیستم‌های Publish-Subscribe



- توزیع جریان‌ها با **DAG** و نمایش‌های publish-subscribe
- **DAG:** Apache Flink, Spark Structured Streaming, Storm, Dataflow
- **Pub-Sub:** Apache Kafka, RabbitMQ, Google Pub/Sub, AWS SNS+SQS
- **Apache Kafka:** یک سیستم pub-sub موازی محبوب
  - تاپل‌ها در یک جریان به یک **topic** منتشر می‌شوند
  - مصرف‌کنندگان به topic **subscribe** می‌کنند

#### 📚 توضیح آموزشی

**Apache Kafka — قلب بسیاری از سیستم‌های Streaming:**

```
مفاهیم اصلی:
Topic:          دسته‌ای از messages (مثل یک queue)
Partition:      هر topic به چند partition تقسیم می‌شود (موازی‌سازی)
Producer:       داده را به topic می‌نویسد
Consumer:       داده را از topic می‌خواند
Consumer Group: چندین consumer که یک topic را parallel می‌خوانند
Offset:         موقعیت در topic
Retention:      Kafka داده را X روز نگه می‌دارد
```

**چرا Kafka انقدر محبوب است؟**
1. **High throughput:** ۱ میلیون message در ثانیه
2. **Persistence:** داده روی دیسک نوشته می‌شود (نه فقط RAM)
3. **Replay:** می‌توانی offset را reset کنی و از اول بخوانی
4. **Multiple Consumers:** چندین سیستم همان داده را می‌خوانند

**معماری مدرن با Kafka:**
```
DB → CDC → Kafka → Flink → Analytics DB
                 → ML Serving
                 → Real-time Dashboard
```

## پایگاه‌های داده گراف

###  ۶۱: مدل داده گراف



- گراف‌ها یک **مدل داده بسیار عمومی** هستند
- مدل ER یک سازمان می‌تواند به عنوان گراف دیده شود:
  - هر موجودیت یک نود است
  - هر رابطه دودویی یک یال است
  - رابطه‌های سه‌گانه می‌توانند به رابطه‌های دودویی مدل شوند

#### 📚 توضیح آموزشی

**چرا به پایگاه داده گراف نیاز داریم؟**

در SQL، جستجوی "دوستان دوستان تا عمق ۵":
```sql
-- برای هر hop یک JOIN لازم است — بسیار کند!
WITH RECURSIVE friends AS (
  SELECT user2 FROM friendships WHERE user1 = 123
  UNION ALL
  SELECT f.user2 FROM friendships f INNER JOIN friends fn ON f.user1 = fn.user2
)
SELECT * FROM friends WHERE depth <= 5;
```

در Neo4j:
```cypher
MATCH (me)-[:FRIEND*1..5]-(result) WHERE me.id = 123 RETURN result
```

فقط یک خط و بسیار سریع‌تر!

**Graph DB مناسب است وقتی:**
- LinkedIn: "چند نفر بین من و این فرد واسطه هستند؟"
- Google Knowledge Graph: روابط بین موجودیت‌های دنیا
- Fraud Detection: شبکه‌های مشکوک تراکنش
- Recommendation Engine: "چون این را دوست داری..."

---

###  ۶۲: مدل داده گراف (ادامه)



- گراف‌ها می‌توانند به عنوان رابطه مدل شوند:
  - `node(ID, label, node_data)`
  - `edge(fromID, toID, label, edge_data)`
- این نمایش خیلی ساده‌گرایانه است
- پایگاه‌های داده گراف مثل **Neo4J** نمای گراف از schema رابطه‌ای ارائه می‌دهند
- زبان‌های پرس‌وجو برای پایگاه‌های داده گراف:
  - پیمایش یال را آسان می‌کنند
  - اجازه می‌دهند الگوریتم‌های کارآمد برای ارزیابی استفاده شوند

#### 📚 توضیح آموزشی

**Property Graph Model:**

```
Node: (Person {name: "Ali", age: 30})
Node: (Person {name: "Sara", age: 28})
Node: (City  {name: "Tehran"})

Edge: Ali -[:KNOWS {since: 2020}]-> Sara
Edge: Ali -[:LIVES_IN]-> Tehran
Edge: Sara -[:LIVES_IN]-> Tehran
```

**زبان‌های پرس‌وجو گراف:**

| زبان | پایگاه داده | مثال |
|------|------------|------|
| Cypher | Neo4j | `MATCH (a)-[:KNOWS]->(b) RETURN b` |
| Gremlin | Apache TinkerPop | `g.V().hasLabel('person').out('knows')` |
| SPARQL | RDF/Semantic Web | برای Linked Data |
| GQL | ISO Standard (2023) | استاندارد جدید |

**چرا بهتر از Relational برای گراف؟**
در پایگاه داده گراف، ساختار داده برای traversal بهینه‌سازی شده است:
- هر نود مستقیماً به یال‌هایش اشاره می‌کند (adjacency list)
- هر hop فقط یک pointer follow است، نه یک JOIN

###  ۶۳: مدل داده گراف — Neo4J



**پرس‌وجو در Neo4J:**

```cypher
-- اساتید و دانشجویانشان در گروه CS
MATCH (i:instructor)<-[:advisor]-(s:student)
WHERE i.dept_name = 'Comp. Sci.'
RETURN i.ID AS ID, i.name AS name, collect(s.name) AS advisees

-- بستار تراگذرا (Transitive Closure) — پیش‌نیازها
MATCH (c1:course)-[:prereq *1..]->(c2:course)
RETURN c1.course_id, c2.course_id
```

#### 📚 توضیح آموزشی

**تحلیل query های Neo4j:**

```cypher
MATCH (i:instructor)<-[:advisor]-(s:student)
```
- `(i:instructor)`: نودهایی با label "instructor"
- `<-[:advisor]-`: یال با label "advisor" که به سمت i است
- `(s:student)`: نودهایی با label "student"

**`[:prereq *1..]` — جادوی گراف:**
```
*1..   → از عمق ۱ تا بی‌نهایت
*1..5  → از عمق ۱ تا ۵
*2..3  → دقیقاً عمق ۲ یا ۳
```

این همان "transitive closure" است که در SQL نیاز به recursive CTE داشت — یک خط در Cypher.

**`collect()` — aggregation در Cypher:**
معادل `array_agg()` در SQL — همه مقادیر را در یک لیست جمع می‌کند.

###  ۶۴: پردازش موازی گراف



- گراف‌های بسیار بزرگ (میلیاردها نود، تریلیون‌ها یال)
  - گراف وب: صفحات وب نودها، هایپرلینک‌ها یال‌ها هستند
  - گراف شبکه اجتماعی: افراد نودها، لینک‌های دوستی/دنبال کردن یال‌ها هستند
- دو رویکرد محبوب:
  - MapReduce و فریمورک‌های جبری
  - **Bulk Synchronous Processing (BSP)**
- BSP سربار بسیار کمتری نسبت به MapReduce دارد
- **سیستم Pregel گوگل** فریمورک BSP را محبوب کرد
- **Apache Giraph:** کپی متن‌باز Pregel
- **Spark GraphX:** API مشابه Pregel

#### 📚 توضیح آموزشی

**چرا MapReduce برای گراف‌ها کند است؟**

الگوریتم‌های گراف مثل PageRank به iteration های متعدد نیاز دارند:
```
MapReduce: هر iteration یک Job = نوشتن به HDFS
           ۵۰ iteration → ۱۰۰ عملیات I/O دیسک!

BSP/Pregel: نودها در RAM می‌مانند
            فقط پیام‌ها بین نودها منتقل می‌شوند
            بسیار کارآمدتر!
```

###  ۶۵: پردازش همگام‌انبوه (BSP)



- هر رأس (نود) گراف داده‌ای (وضعیت) با خود دارد
- رأس‌ها بین ماشین‌های متعدد تقسیم می‌شوند و وضعیت نود در حافظه نگه داشته می‌شود
- برنامه‌نویسان متدهایی برای هر نود فراهم می‌کنند — می‌تواند پیام بفرستد یا دریافت کند
- **محاسبات از چندین superstep تشکیل می‌شود:**
  - در هر superstep: نودها پیام‌های دریافتی را پردازش می‌کنند
  - وضعیت خود را به‌روزرسانی می‌کنند
  - پیام‌های بیشتر می‌فرستند یا **رأی به توقف** می‌دهند
- محاسبات زمانی پایان می‌یابد که همه نودها رأی به توقف دهند و پیام معلقی وجود نداشته باشد

#### 📚 توضیح آموزشی

**مدل "Think Like a Vertex":**

```
هر نود در هر superstep:
  1. پیام‌های رسیده از همسایگان را می‌خوانم
  2. وضعیتم را update می‌کنم
  3. به همسایگانم پیام می‌فرستم (یا halt می‌کنم)
```

**PageRank در Pregel:**
```java
void compute(List<Double> messages) {
  double sum = 0;
  for (double m : messages) sum += m;

  // وضعیت = pagerank این نود
  double newRank = 0.15/N + 0.85 * sum;
  setValue(newRank);

  // برای هر همسایه خروجی
  for (Edge e : getOutEdges()) {
    sendMessage(e.target, newRank / numOutEdges());
  }
}
```

**Vote to Halt:**
وقتی یک نود کارش تمام شد، halt می‌کند. اگر پیام جدید برسد، دوباره فعال می‌شود.
کل محاسبه وقتی **همه نودها halt کرده‌اند AND هیچ پیام pending نیست** تمام می‌شود.

**Superstep Synchronization:**
همه نودها پیام‌های superstep k را همزمان پردازش می‌کنند. نتایج در superstep k+1 دیده می‌شوند. این barrier synchronization ساده است اما کارآمد.
