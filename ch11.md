## مرور کلی

###  ۳: مرور کلی



- **تحلیل داده:** پردازش داده برای استنتاج الگوها، همبستگی‌ها یا مدل‌های پیش‌بینی
- عمدتاً برای تصمیم‌گیری‌های تجاری استفاده می‌شود:
  - **برای هر مشتری به صورت جداگانه:** مثلاً چه محصولی برای خرید پیشنهاد شود
  - **در سطح همه مشتریان:** مثلاً چه محصولاتی تولید/انبار شوند و به چه مقداری
- امروزه برای کسب‌وکارها حیاتی است

#### 📚 توضیح آموزشی

**تحلیل داده چیست؟**

تحلیل داده (Data Analytics) فرایند تبدیل داده‌های خام به اطلاعات قابل استفاده برای تصمیم‌گیری است. می‌توان آن را در سه سطح دید:

| سطح | مثال | روش |
|-----|------|-----|
| توصیفی (Descriptive) | فروش دیروز چقدر بود؟ | گزارش‌گیری، داشبورد |
| تشخیصی (Diagnostic) | چرا فروش کاهش یافت؟ | کاوش داده، OLAP |
| پیش‌بینی (Predictive) | فروش هفته آینده چقدر خواهد بود؟ | Machine Learning |

**کاربرد واقعی:** وقتی Netflix یک سریال پیشنهاد می‌دهد، نتیجه تحلیل داده است — ترکیب تاریخچه تماشای شما و الگوهای مشابه در میلیون‌ها کاربر دیگر.

---

###  ۴: مرور کلی (ادامه)



**مراحل رایج در تحلیل داده:**

1. **جمع‌آوری داده از منابع مختلف** در یک مکان
   - انبارهای داده نیز داده‌ها را در یک schema یکپارچه ادغام می‌کنند
   - داده اغلب باید استخراج، تبدیل و بارگذاری شود: **ETL** یا **ELT**
2. **تولید گزارش‌ها و خلاصه‌های تجمیعی**
   - داشبوردها با نمودارها و گزارش‌های گرافیکی
   - سیستم‌های OLAP اجازه پرس‌وجوی تعاملی می‌دهند
   - تحلیل آماری با ابزارهایی مانند R، SAS، SPSS
3. **ساخت مدل‌های پیش‌بینی** و استفاده از آن‌ها در تصمیم‌گیری

#### 📚 توضیح آموزشی

**ETL در مقابل ELT — تفاوت چیست؟**

```
ETL (Extract → Transform → Load):
منبع داده → [استخراج] → [تبدیل] → [بارگذاری] → انبار داده
```

```
ELT (Extract → Load → Transform):
منبع داده → [استخراج] → [بارگذاری] → انبار داده → [تبدیل]
```

- **ETL (رویکرد سنتی):** داده را قبل از ورود به انبار پاکسازی و تبدیل کن. مناسب وقتی انبار داده کوچک‌تر و منابع محدود است.
- **ELT (رویکرد مدرن):** ابتدا همه چیز را بارگذاری کن، سپس با قدرت پردازش انبار (مثل Snowflake یا BigQuery) تبدیل کن. مناسب برای Cloud Data Warehouses.

**Pipeline داده نمونه:**
```
CRM (Salesforce) ──┐
ERP (SAP)         ──┤── ETL/ELT ──► Data Warehouse ──► OLAP ──► Dashboard
وب‌سایت           ──┤                                            ┕──► ML Model
شبکه اجتماعی    ──┘
```

###  ۵: مرور کلی (ادامه)



**مدل‌های پیش‌بینی امروزه به طور گسترده استفاده می‌شوند:**

- استفاده از ویژگی‌های پروفایل مشتری (درآمد، سن، جنسیت، تحصیلات، شغل) و تاریخچه گذشته برای پیش‌بینی احتمال نکول وام
- استفاده از تاریخچه فروش (بر اساس فصل) برای پیش‌بینی فروش آینده

**سایر نمونه‌های تصمیم‌گیری تجاری:**
- چه اقلامی انبار شوند؟
- چه حق بیمه‌ای تعیین شود؟
- تبلیغات به چه کسانی ارسال شوند؟

#### 📚 توضیح آموزشی

**Credit Scoring — مثال کلاسیک:**

بانک‌ها از مدل‌های پیش‌بینی برای ارزیابی ریسک اعتباری استفاده می‌کنند. ورودی‌ها:
- سن، درآمد، سابقه اشتغال
- تاریخچه پرداخت وام‌های قبلی
- نسبت بدهی به درآمد

خروجی: امتیاز اعتباری (مثلاً FICO Score)

**چرخه پیش‌بینی در کسب‌وکار:**
```
داده تاریخی ──► مدل ML ──► پیش‌بینی ──► تصمیم تجاری ──► داده جدید
     ↑                                                          │
     └──────────────────── بازخورد ◄──────────────────────────┘
```

این حلقه بازخورد مدل را به مرور زمان بهبود می‌دهد.

###  ۶: مرور کلی (ادامه)



- **یادگیری ماشین (Machine Learning):** کلید یافتن الگوها در داده و پیش‌بینی
- **داده‌کاوی (Data Mining):** تکنیک‌های یادگیری ماشین روی حجم بسیار زیادی از داده
- **هوش تجاری (Business Intelligence / BI):** مترادف با تحلیل داده
- **پشتیبانی تصمیم (Decision Support):** تمرکز بر گزارش‌گیری و تجمیع

#### 📚 توضیح آموزشی

**نقشه جامع اصطلاحات:**

```
هوش تجاری (BI)
    ├── گزارش‌گیری و داشبورد
    ├── OLAP (پرس‌وجوی تعاملی)
    └── تحلیل پیشرفته
            ├── آمار (R, SAS, SPSS)
            └── داده‌کاوی / ML
                    ├── یادگیری ماشین (ML)
                    └── یادگیری عمیق (Deep Learning)
```

**تفاوت ML و Data Mining:**
- **ML:** بیشتر بر الگوریتم‌ها و مدل‌ها تمرکز دارد
- **Data Mining:** بر کشف الگوهای مفید از داده‌های بزرگ تمرکز دارد
- در عمل این دو به شدت همپوشانی دارند

## انبار داده

###  ۸: انبار داده



- منابع داده اغلب فقط داده‌های جاری را ذخیره می‌کنند نه تاریخی
- تصمیم‌گیری سازمانی نیازمند دیدی یکپارچه از همه داده‌های سازمانی شامل داده‌های تاریخی است
- **انبار داده:** مخزن (آرشیو) اطلاعات جمع‌آوری‌شده از منابع مختلف، ذخیره‌شده تحت یک schema یکپارچه در یک سایت
- پرس‌وجو را بسیار ساده می‌کند، مطالعه روندهای تاریخی را ممکن می‌سازد
- بار پرس‌وجوی پشتیبانی تصمیم را از سیستم‌های پردازش تراکنش جدا می‌کند

#### 📚 توضیح آموزشی

**چرا انبار داده جدا از پایگاه داده عملیاتی؟**

| ویژگی | پایگاه داده عملیاتی (OLTP) | انبار داده (OLAP) |
|-------|--------------------------|-----------------|
| هدف | ثبت تراکنش‌های روزانه | تحلیل و گزارش‌گیری |
| عملیات | Insert, Update, Delete | عمدتاً Select |
| بازه زمانی | داده جاری (چند ماه) | داده تاریخی (سال‌ها) |
| حجم هر query | کم (چند رکورد) | زیاد (میلیون‌ها رکورد) |
| نرمال‌سازی | بالا (3NF) | کم (Star Schema) |
| کاربران | کارمندان اجرایی | تحلیل‌گران داده |

**مثال واقعی:** اگر Amazon از همان پایگاه داده‌ای که سفارش‌ها را ثبت می‌کند برای تحلیل فروش هم استفاده می‌کرد، query های سنگین تحلیلی سرعت ثبت سفارش را به شدت کاهش می‌داد — فاجعه!

---

###  ۱۰: مسائل طراحی



**چه زمانی و چگونه داده جمع‌آوری شود:**

- **معماری مبدأ-محور:** منابع داده اطلاعات جدید را به انبار ارسال می‌کنند (مداوم یا دوره‌ای)
- **معماری مقصد-محور:** انبار به صورت دوره‌ای اطلاعات جدید را از منابع درخواست می‌کند
- همگام‌سازی دقیق با two-phase commit معمولاً خیلی گران است
- معمولاً داشتن داده‌های کمی قدیمی در انبار قابل‌قبول است
- داده‌ها به صورت دوره‌ای از سیستم‌های OLTP دریافت می‌شوند
- **یکپارچه‌سازی schema**

#### 📚 توضیح آموزشی

**Change Data Capture (CDC) — رویکرد مدرن:**

به جای کپی کامل جدول، فقط تغییرات را منتقل کن:
```
رکورد تغییر یافته در MySQL → Debezium (CDC Tool) → Kafka → Data Warehouse
```

**مقایسه رویکردها:**

| روش | مزیت | معایب |
|-----|-------|-------|
| Full Load | ساده | کند، فشار زیاد روی OLTP |
| Incremental Load | سریع‌تر | نیاز به timestamp یا کلید تغییر |
| CDC | Real-time | پیچیدگی بیشتر |
| Event Streaming | لحظه‌ای | نیاز به Kafka/Message Queue |

**تأخیر مجاز:**
- گزارش مالی روزانه: تأخیر یک شب قابل‌قبول است
- تشخیص تقلب: باید real-time باشد
- پیشنهاد محصول: چند دقیقه قابل‌قبول است

###  ۱۱: مسائل بیشتر طراحی انبار



- **تبدیل داده و پاکسازی داده:** مثلاً تصحیح آدرس‌های اشتباه، ادغام لیست‌های آدرس از منابع مختلف
- **نحوه انتشار به‌روزرسانی‌ها:** انبار ممکن است نمای (View) مادی‌شده‌ای از schema منابع باشد
- **چه داده‌ای خلاصه شود:**
  - داده خام ممکن است برای ذخیره آنلاین خیلی بزرگ باشد
  - مقادیر تجمیعی (جمع‌ها/جزئی‌جمع‌ها) اغلب کافی است
  - query های روی داده خام اغلب توسط query optimizer به استفاده از مقادیر تجمیعی تبدیل می‌شوند

#### 📚 توضیح آموزشی

**Data Quality — چالش اصلی:**

در عمل، ۸۰٪ زمان پروژه‌های داده صرف پاکسازی داده می‌شود!

مشکلات رایج:
```
• نام‌های مختلف برای یک چیز: "تهران" / "Tehran" / "TEHRAN"
• مقادیر گمشده (NULL)
• فرمت‌های متفاوت: "۱۴۰۲/۰۱/۰۱" vs "2023-03-21"
• داده‌های تکراری از منابع مختلف
• مقادیر نامعتبر: سن منفی، ایمیل بدون @
```

**Materialized Views در انبار:**
یک aggregate از قبل محاسبه‌شده است. مثلاً:
```sql
-- به جای اینکه هر بار این query سنگین اجرا شود:
SELECT region, SUM(sales) FROM transactions GROUP BY region

-- View مادی‌شده نگه می‌دارد:
CREATE MATERIALIZED VIEW regional_sales AS
SELECT region, SUM(sales) FROM transactions GROUP BY region;
```
Query Optimizer وقتی ببیند می‌توانی از view استفاده کنی، query اصلی را redirect می‌کند.

###  ۱۲: داده چندبُعدی و Schema انبار



**داده در انبارها معمولاً به دو نوع تقسیم می‌شود:**

- **جداول حقیقت (Fact Tables):** بزرگ
  - مثال: `sales(item_id, store_id, customer_id, date, number, price)`
- **جداول ابعاد (Dimension Tables):** نسبتاً کوچک
  - اطلاعات اضافی درباره فروشگاه‌ها، اقلام و ...

**ویژگی‌های جدول حقیقت:**
- **ویژگی‌های معیار (Measure Attributes):** یک مقدار را اندازه می‌گیرند و می‌توان روی‌شان تجمیع کرد (مثل `number` یا `price`)
- **ویژگی‌های بُعد (Dimension Attributes):** ابعادی که معیارها از آن زاویه دیده می‌شوند (مثل `item_id`, `color`, `size`)

#### 📚 توضیح آموزشی

**Star Schema — معماری اصلی انبار:**

```
                    ┌─────────────┐
                    │ dim_date    │
                    │ date_id (PK)│
                    │ day, month  │
                    │ quarter,year│
                    └──────┬──────┘
                           │
┌─────────────┐    ┌───────┴──────────┐    ┌─────────────┐
│ dim_item    │    │   fact_sales      │    │ dim_store   │
│ item_id (PK)│◄───│ item_id (FK)     │───►│ store_id(PK)│
│ name, color │    │ store_id (FK)    │    │ city, region│
│ category    │    │ customer_id (FK) │    │ manager     │
└─────────────┘    │ date_id (FK)     │    └─────────────┘
                   │ quantity (Measure)│
                   │ price (Measure)   │
                   └──────────────────┘
```

**چرا Star Schema؟**
- JOIN ساده: جدول حقیقت + چند جدول بُعد
- بهینه برای aggregate queries
- قابل فهم برای business users

**Snowflake Schema:** وقتی جداول بُعد هم normalized می‌شوند:
```
dim_item → dim_category → dim_department
```
پیچیده‌تر اما فضای کمتری می‌گیرد.

###  ۱۴: داده چندبُعدی و Schema (ادامه)



- **Star Schema:** نتیجه schema است
- **Snowflake Schema:** سطوح متعدد جداول بُعد
- ممکن است چندین جدول حقیقت وجود داشته باشد
- معمولاً: جدول حقیقت با جداول بُعد join، سپس group-by روی ویژگی‌های بُعد، سپس تجمیع
- **Data Lake:** مخازنی که داده را در فرمت‌های متعدد بدون یکپارچه‌سازی schema ذخیره می‌کنند
  - تلاش کمتر ابتدایی اما تلاش بیشتر هنگام پرس‌وجو

#### 📚 توضیح آموزشی

**Data Warehouse در مقابل Data Lake:**

| | Data Warehouse | Data Lake |
|--|---------------|-----------|
| Schema | از پیش تعریف‌شده | Schema-on-read |
| داده | ساختارمند | هر نوعی |
| کاربران | تحلیل‌گران BI | Data Scientists |
| ابزار | SQL, Tableau | Python, Spark, ML |
| مثال | Snowflake, Redshift | AWS S3 + Athena |

**Data Lakehouse (ترکیب مدرن):**
رویکرد جدیدی که بهترین هر دو را ترکیب می‌کند:
```
Data Lake (ذخیره ارزان) + Transaction Support + SQL Interface
مثال: Databricks Delta Lake, Apache Iceberg, Apache Hudi
```

**الگوی رایج query در انبار:**
```sql
SELECT s.region, i.category, d.quarter, SUM(f.quantity * f.price) as revenue
FROM fact_sales f
JOIN dim_store s ON f.store_id = s.store_id
JOIN dim_item i ON f.item_id = i.item_id
JOIN dim_date d ON f.date_id = d.date_id
WHERE d.year = 2024
GROUP BY s.region, i.category, d.quarter
ORDER BY revenue DESC;
```

###  ۱۵: پشتیبانی پایگاه داده از انبارها



- داده در انبارها معمولاً فقط اضافه می‌شود، به‌روز نمی‌شود (append-only)
- می‌توان از سربار کنترل همروندی اجتناب کرد
- انبارهای داده اغلب از **ذخیره‌سازی ستون‌محور (Column-Oriented Storage)** استفاده می‌کنند
  - مقادیر `item_id` به عنوان یک آرایه ذخیره می‌شوند، `store_id` به عنوان آرایه دیگر، و ...
  - آرایه‌ها فشرده می‌شوند — کاهش چشمگیر در ذخیره، I/O و حافظه
  - query ها فقط ستون‌هایی که نیاز دارند را واکشی می‌کنند
- انبارهای داده اغلب از زیرساخت ذخیره‌سازی و پردازش query موازی استفاده می‌کنند

#### 📚 توضیح آموزشی

**ذخیره‌سازی ردیف‌محور در مقابل ستون‌محور:**

**ردیف‌محور (Row-Oriented) — مثل PostgreSQL:**
```
[Ali | 30 | Tehran | Engineer] [Sara | 28 | Mashhad | Doctor] ...
```
وقتی یک ردیف کامل می‌خواهی سریع است، اما وقتی باید یک ستون از میلیون‌ها ردیف بخوانی باید همه چیز را بخوانی.

**ستون‌محور (Column-Oriented) — مثل Parquet, Snowflake:**
```
name:  [Ali, Sara, ...]
age:   [30, 28, ...]
city:  [Tehran, Mashhad, ...]
job:   [Engineer, Doctor, ...]
```

مزایا:
- **فشرده‌سازی فوق‌العاده:** مقادیر یکسان در کنار هم → Run-Length Encoding عالی
- **I/O کم:** برای `SELECT AVG(age)` فقط ستون age می‌خوانیم
- **SIMD operations:** پردازنده می‌تواند روی آرایه‌ها به صورت موازی کار کند

**مثال فشرده‌سازی:**
```
[Engineer, Engineer, Engineer, Doctor, Doctor, Doctor, Doctor, Doctor]
→ فشرده: [(Engineer, 3), (Doctor, 5)]
→ ۸ رکورد → ۲ رکورد!
```

## پردازش تحلیلی آنلاین

###  ۱۷: تحلیل داده و OLAP



**پردازش تحلیلی آنلاین (OLAP):**
- تحلیل تعاملی داده، اجازه می‌دهد داده به صورت آنلاین (با تأخیر ناچیز) خلاصه شود و از زوایای مختلف دیده شود
- رابطه نمونه برای توضیح مفاهیم OLAP:
  - `sales(item_name, color, clothes_size, quantity)`

#### 📚 توضیح آموزشی

**OLAP چیست؟**

OLAP ابزار تحلیل‌گر است. به جای نوشتن SQL پیچیده، می‌توانی:
- بکشی و رها کنی (Drag & Drop)
- ابعاد را عوض کنی
- از کلی به جزئی بروی یا برعکس

**مقایسه OLTP و OLAP:**
```
OLTP: "سفارش شماره ۱۲۳۴ را ثبت کن"  → چند ردیف
OLAP: "فروش کل کفش در تهران در Q2"   → میلیون‌ها ردیف
```

ابزارهای محبوب OLAP: Tableau, Power BI, Looker, Apache Superset

###  ۱۹: جدول متقاطع (Cross Tabulation)



- **جدول متقاطع (Cross-Tab / Pivot Table):** مقادیر یک ویژگی بُعدی سطرها را تشکیل می‌دهند، ویژگی بُعدی دیگر ستون‌ها را
- سایر ویژگی‌های بُعدی در بالا فهرست می‌شوند
- مقادیر در خانه‌های فردی (تجمیع از) مقادیر ویژگی‌های بُعدی هستند

#### 📚 توضیح آموزشی

**مثال Pivot Table:**

داده خام:
```
item_name | color  | quantity
----------|--------|--------
shirt     | dark   | 2
shirt     | white  | 3
pants     | dark   | 1
pants     | pastel | 5
```

Cross-Tab (Pivot):
```
           | dark | pastel | white | ALL
-----------|------|--------|-------|-----
shirt      |  2   |   0    |   3   |  5
pants      |  1   |   5    |   0   |  6
ALL        |  3   |   5    |   3   | 11
```

سطر/ستون "ALL" مجموع است — این همان `GROUP BY CUBE` در SQL است!

---

###  ۲۰: مکعب داده (Data Cube)



- **مکعب داده:** تعمیم چندبُعدی جدول متقاطع
- می‌تواند n بُعد داشته باشد
- Cross-tab ها می‌توانند به عنوان نماهایی روی مکعب داده استفاده شوند

#### 📚 توضیح آموزشی

**Data Cube در ذهن:**

تصور کن فروش را از سه زاویه می‌بینی:
```
        ┌─────────────────────┐
       /│    ماه: مارس        │
      / │                     │
     /  │  item   │ dark │ white
    /   ├─────────┼──────┼──────
   /    │ shirt   │  5   │  3
  /     │ pants   │  2   │  8
 ┌──────┴─────────────────────┐
/      ماه: فوریه             /
```

هر برش (slice) یک Cross-Tab است. هر مکعب یک بُعد اضافه‌تر دارد.

**چرا مهم است؟**
یک مکعب ۱۰ بُعدی = ۲¹⁰ = ۱۰۲۴ ترکیب مختلف برای group-by. محاسبه همه آن‌ها از پیش (pre-computation) امکان پرس‌وجوی بدون تأخیر را می‌دهد.

---

###  ۲۱: عملیات OLAP



- **Pivoting:** تغییر ابعاد استفاده‌شده در cross-tab (مثلاً جابجایی رنگ‌ها به ستون‌ها)
- **Slicing:** ایجاد cross-tab برای مقادیر ثابت فقط (مثلاً رنگ = سفید و سایز = کوچک)
- **Dicing:** وقتی مقادیر چندین بُعد ثابت می‌شوند
- **Roll-up:** حرکت از دانه‌بندی ریزتر به درشت‌تر (مثلاً از روزانه به ماهانه)
- **Drill-down:** عملیات معکوس — از دانه‌بندی درشت به ریز

#### 📚 توضیح آموزشی

**۵ عملیات اصلی OLAP با مثال:**

```
Pivot: جابجایی محورها
قبل: ردیف=رنگ، ستون=سایز
بعد: ردیف=سایز، ستون=رنگ

Slice: برش یک بُعد
"فقط محصولات رنگ قرمز نشانم بده"

Dice: برش چند بُعد
"محصولات قرمز، سایز M و L نشانم بده"

Roll-up: خلاصه‌سازی
فروش روزانه → فروش هفتگی → فروش ماهانه → فروش سالانه

Drill-down: جزئیات بیشتر
فروش سالانه → فروش ماهانه → فروش روزانه → فروش ساعتی
```

**در Tableau:**
این عملیات‌ها با drag & drop انجام می‌شوند — بدون نوشتن SQL!

###  ۲۲: سلسله‌مراتب روی ابعاد



- **سلسله‌مراتب روی ویژگی‌های بُعدی:** اجازه می‌دهند ابعاد در سطوح مختلف جزئیات دیده شوند
- مثال: بُعد datetime می‌تواند بر اساس ساعت، روز، روز هفته، ماه، فصل یا سال تجمیع شود

#### 📚 توضیح آموزشی

**سلسله‌مراتب زمانی:**
```
سال
 └── فصل (Q1, Q2, Q3, Q4)
      └── ماه (ژانویه، فوریه، ...)
           └── هفته
                └── روز
                     └── ساعت
                          └── دقیقه
```

**سلسله‌مراتب جغرافیایی:**
```
کشور → استان → شهر → ناحیه → فروشگاه
```

**سلسله‌مراتب محصول:**
```
دپارتمان → دسته‌بندی → زیر‌دسته → محصول → SKU
```

این سلسله‌مراتب در **Snowflake Schema** به صورت جداول مجزا ذخیره می‌شوند و Roll-up/Drill-down را طبیعی می‌کنند.

###  ۲۵–۲۶: OLAP در SQL — عملیات Pivot



```sql
SELECT *
FROM sales
PIVOT (
    SUM(quantity)
    FOR color IN ('dark', 'pastel', 'white')
)
ORDER BY item_name;
```

#### 📚 توضیح آموزشی

**PIVOT در SQL — تبدیل ردیف به ستون:**

```sql
-- بدون PIVOT (نتیجه ردیفی):
SELECT item_name, color, SUM(quantity)
FROM sales
GROUP BY item_name, color;

-- نتیجه:
-- shirt | dark   | 2
-- shirt | white  | 3

-- با PIVOT (نتیجه جدولی):
-- item_name | dark | pastel | white
-- shirt     |  2   |   0    |   3
```

**PIVOT در پایگاه‌های مختلف:**
```sql
-- Oracle/SQL Server: دارای PIVOT
SELECT * FROM sales PIVOT (SUM(quantity) FOR color IN ('dark','white'));

-- PostgreSQL: از FILTER استفاده کن
SELECT item_name,
  SUM(quantity) FILTER (WHERE color = 'dark') AS dark,
  SUM(quantity) FILTER (WHERE color = 'white') AS white
FROM sales GROUP BY item_name;
```

###  ۲۷: عملیات Cube



```sql
SELECT item_name, color, size, SUM(number)
FROM sales
GROUP BY CUBE(item_name, color, size)
```

این query union هشت گروه‌بندی مختلف را محاسبه می‌کند:
```
{ (item_name, color, size), (item_name, color),
  (item_name, size),        (color, size),
  (item_name),              (color),
  (size),                   ( ) }
```

#### 📚 توضیح آموزشی

**GROUP BY CUBE — همه ترکیب‌ها:**

با ۳ ویژگی، ۲³ = ۸ ترکیب group-by وجود دارد. CUBE همه آن‌ها را یکجا محاسبه می‌کند.

```sql
-- معادل دستی CUBE:
SELECT item_name, color, size, SUM(number) FROM sales GROUP BY item_name, color, size
UNION ALL
SELECT item_name, color, NULL, SUM(number) FROM sales GROUP BY item_name, color
UNION ALL
SELECT item_name, NULL, size, SUM(number) FROM sales GROUP BY item_name, size
-- ... و ۵ گروه‌بندی دیگر
```

**CUBE vs ROLLUP:**
```sql
CUBE(a, b, c)   → همه ۲ⁿ زیرمجموعه‌ها
ROLLUP(a, b, c) → فقط {(a,b,c), (a,b), (a), ()} — سلسله‌مراتب!
```

**کاربرد ROLLUP:**
```sql
-- گزارش سلسله‌مراتبی: دسته → محصول
SELECT category, item_name, SUM(number)
FROM sales, itemcategory
WHERE sales.item_name = itemcategory.item_name
GROUP BY ROLLUP(category, item_name);
-- نتیجه: جمع کل هر دسته + جمع هر محصول در آن دسته
```

###  ۳۲: پیاده‌سازی OLAP



- **MOLAP (Multidimensional OLAP):** داده مکعب را در آرایه‌های چندبُعدی در حافظه ذخیره می‌کند
- **ROLAP (Relational OLAP):** فقط از ویژگی‌های پایگاه داده رابطه‌ای استفاده می‌کند
- **HOLAP (Hybrid OLAP):** برخی خلاصه‌ها در حافظه، داده پایه در پایگاه داده رابطه‌ای

#### 📚 توضیح آموزشی

**مقایسه سه رویکرد:**

| | MOLAP | ROLAP | HOLAP |
|--|-------|-------|-------|
| سرعت | بسیار سریع | متوسط | خوب |
| فضا | زیاد | کم | متوسط |
| انعطاف | کم | زیاد | متوسط |
| حجم داده | کوچک تا متوسط | هر حجمی | متوسط تا بزرگ |
| مثال | Essbase (Oracle) | Hive, BigQuery | SSAS (Microsoft) |

**رویکرد مدرن:**
امروزه اکثر سیستم‌های OLAP از columnar storage (مثل Parquet در Spark) استفاده می‌کنند که سرعت MOLAP را با انعطاف ROLAP ترکیب می‌کند.

###  ۳۳: پیاده‌سازی OLAP (ادامه)



- سیستم‌های اولیه OLAP همه aggregateهای ممکن را از پیش محاسبه می‌کردند
- این می‌تواند الزامات فضا و زمان بسیار بالایی داشته باشد (۲ⁿ ترکیب)
- کافی است برخی aggregate ها از پیش محاسبه شوند
- می‌توان aggregate روی (item_name, color) را از (item_name, color, size) محاسبه کرد
- چندین بهینه‌سازی برای محاسبه چندین aggregate وجود دارد

#### 📚 توضیح آموزشی

**Aggregate Lattice — گراف وابستگی:**

```
           {}
          / | \
         a  b  c
        /\  |  /\
       ab ac bc
          \  |  /
           abc
```

اگر aggregate (a,b,c) را محاسبه کرده‌ایم، می‌توانیم (a,b) را با یک pass اضافه روی آن به دست بیاوریم — نه از صفر!

**Aggregate ‌های غیرتجزیه‌پذیر:**
بعضی aggregate ها قابل تجزیه نیستند:
- Median: نمی‌توانی median کل را از median بخش‌ها به دست بیاوری
- Percentile, Mode: همین مشکل را دارند
- Sum, Count, Max, Min: کاملاً تجزیه‌پذیرند ✓

## داده‌کاوی

###  ۳۶: داده‌کاوی



- **داده‌کاوی:** فرایند نیمه‌خودکار تحلیل پایگاه داده‌های بزرگ برای یافتن الگوهای مفید
- اهداف مشابه یادگیری ماشین، اما روی حجم بسیار زیاد داده
- بخشی از حوزه بزرگ‌تر کشف دانش در پایگاه داده (KDD)
- برخی دانش‌ها می‌توانند به عنوان قوانین نمایش داده شوند
- دانش با اعمال تکنیک‌های ML روی نمونه‌های گذشته کشف می‌شود تا مدلی شکل گیرد

#### 📚 توضیح آموزشی

**KDD (Knowledge Discovery in Databases) — فرایند کامل:**

```
داده خام
   ↓ [انتخاب]
زیرمجموعه مرتبط
   ↓ [پیش‌پردازش]
داده تمیز
   ↓ [تبدیل]
داده در قالب مناسب
   ↓ [داده‌کاوی]
الگوها و مدل‌ها
   ↓ [تفسیر/ارزیابی]
دانش
```

**داده‌کاوی در مقابل ML:**
```
ML تمرکز دارد بر: الگوریتم‌ها، نظریه
Data Mining تمرکز دارد بر: داده واقعی بزرگ، قابل تفسیر بودن، اتوماسیون
```

###  ۳۷: انواع وظایف داده‌کاوی



**پیش‌بینی بر اساس تاریخچه گذشته:**
- پیش‌بینی ریسک اعتباری متقاضی کارت اعتباری
- پیش‌بینی تقلبی بودن الگوی استفاده از کارت تلفن

**نمونه‌هایی از مکانیزم‌های پیش‌بینی:**
- **طبقه‌بندی (Classification):** اقلام به یکی از چندین کلاس تعلق دارند؛ با داده‌های آموزشی، کلاس نمونه‌های جدید پیش‌بینی می‌شود
- **فرمول‌های رگرسیون (Regression):** با داشتن مجموعه‌ای از نگاشت‌های یک تابع ناشناخته، نتیجه تابع برای پارامتر جدید پیش‌بینی می‌شود

#### 📚 توضیح آموزشی

**Classification در مقابل Regression:**

| | Classification | Regression |
|--|---------------|------------|
| خروجی | کلاس/برچسب | عدد پیوسته |
| مثال | spam/not-spam | قیمت خانه |
| الگوریتم‌ها | Decision Tree, SVM, Neural Net | Linear, Polynomial |
| معیار ارزیابی | Accuracy, F1-Score | RMSE, MAE |

**مثال ملموس:**
```
Classification: "آیا این ایمیل spam است؟"  → بله / خیر
Regression:     "این خانه چقدر ارزش دارد؟" → ۵۰۰,۰۰۰ دلار
```

**Feature Engineering — مهم‌ترین مرحله:**
قبل از آموزش مدل، باید ویژگی‌های مناسب بسازی:
```
ایمیل خام → [تعداد لینک, طول متن, وجود کلمات spam, نسبت حروف بزرگ] → مدل
```

###  ۳۸: داده‌کاوی (ادامه) — الگوهای توصیفی



**الگوهای توصیفی:**

- **انجمن‌ها (Associations):**
  - یافتن کتاب‌هایی که اغلب توسط مشتریان «مشابه» خریده می‌شوند
  - انجمن‌ها ممکن است به عنوان اولین قدم در تشخیص علیت استفاده شوند
  
- **خوشه‌ها (Clusters):**
  - تشخیص موارد حصبه در یک ناحیه اطراف یک چاه آلوده
  - تشخیص خوشه‌ها در تشخیص اپیدمی‌ها اهمیت دارد

#### 📚 توضیح آموزشی

**Descriptive vs Predictive Mining:**

```
Predictive: "بر اساس X، Y را پیش‌بینی کن"
    مثال: بر اساس تاریخچه خرید، احتمال ریزش مشتری را پیش‌بینی کن

Descriptive: "الگوهای موجود در داده را پیدا کن"
    مثال: کدام محصولات با هم خریده می‌شوند؟ (Association)
    مثال: مشتریان را بر اساس رفتار دسته‌بندی کن (Clustering)
```

**مثال واقعی Association:**
Amazon "Customers who bought this also bought..." — این نتیجه Association Mining است، نه پیش‌بینی دستی!

**مثال واقعی Clustering:**
بانک مرکزی برای تشخیص پول‌شویی: معاملات را cluster می‌کند؛ معاملاتی که در هیچ cluster عادی جا نمی‌گیرند مشکوکند.

###  ۳۹: قوانین طبقه‌بندی



- قوانین طبقه‌بندی به تخصیص اشیاء جدید به کلاس‌ها کمک می‌کنند
- مثال: طبقه‌بندی متقاضی بیمه خودرو به کم‌ریسک، متوسط‌ریسک یا پرریسک

```
∀ person P,  P.degree = masters AND P.income > 75,000
                → P.credit = excellent

∀ person P,  P.degree = bachelors AND
             (P.income > 25,000 AND P.income < 75,000)
                → P.credit = good
```

- قوانین لزوماً دقیق نیستند، ممکن است برخی طبقه‌بندی‌ها اشتباه باشند
- قوانین طبقه‌بندی می‌توانند به صورت درخت تصمیم نشان داده شوند

#### 📚 توضیح آموزشی

**از قوانین IF-THEN به مدل:**

قوانین طبقه‌بندی قابل تفسیر هستند — می‌توانی بفهمی چرا مدل یک تصمیم گرفت.
این "Explainability" در حوزه‌هایی مثل اعتبارسنجی، پزشکی و حقوق بسیار مهم است.

**قوانین دستی در مقابل یادگیری‌شده:**
```
دستی (Expert System):
    IF income > 100K AND age > 30 → low_risk

یادگیری‌شده (Decision Tree):
    مدل از ۱ میلیون رکورد گذشته یاد می‌گیرد چه ترکیبی بهترین predictor است
```

###  ۴۱: درخت‌های تصمیم



- هر گره داخلی درخت داده را بر اساس یک ویژگی تقسیم‌بندی تقسیم می‌کند
- **گره برگ:** همه (یا اکثر) اقلام به یک کلاس تعلق دارند، یا همه ویژگی‌ها بررسی شده‌اند
- از بالا درخت را پیمایش کن تا پیش‌بینی انجام دهی

#### 📚 توضیح آموزشی

**ساختار درخت تصمیم:**

```
               [income > 50K?]
              /               \
           بله                خیر
          /                    \
  [age > 30?]              [degree?]
   /      \                 /      \
 بله      خیر          masters    other
  |        |              |          |
good     medium        excellent    bad
```

**مزایا:**
- قابل تفسیر توسط انسان
- نیازی به normalization ندارد
- ویژگی‌های نامهم را نادیده می‌گیرد

**معایب:**
- Overfitting — درخت خیلی عمیق → حفظ کردن داده آموزشی، نه یادگیری
- بی‌ثبات — تغییر کوچک در داده → درخت کاملاً متفاوت

**راه‌حل Overfitting:**
- Pruning: قطع شاخه‌های بیش از حد جزئی
- Random Forest: ترکیب صدها درخت با bootstrap sampling

###  ۴۲–۴۳: طبقه‌بند‌های بیزی



```
p(cⱼ | d) = p(d | cⱼ) × p(cⱼ) / p(d)
```

- `p(cⱼ | d)`: احتمال اینکه نمونه d در کلاس cⱼ باشد
- `p(d | cⱼ)`: احتمال تولید نمونه d با فرض کلاس cⱼ
- `p(cⱼ)`: احتمال وقوع کلاس cⱼ
- `p(d)`: احتمال وقوع نمونه d

**Naïve Bayes:** فرض می‌کند ویژگی‌ها توزیع مستقلی دارند:
```
p(d | cⱼ) = p(d₁ | cⱼ) × p(d₂ | cⱼ) × ... × p(dₙ | cⱼ)
```

#### 📚 توضیح آموزشی

**Naïve Bayes برای تشخیص spam:**

```
کلاس‌ها: spam, not-spam
ویژگی‌ها: کلمات در ایمیل

آموزش:
  p("free" | spam) = 0.8
  p("meeting" | spam) = 0.02
  p("free" | not-spam) = 0.05
  p("meeting" | not-spam) = 0.6
  p(spam) = 0.3

پیش‌بینی برای ایمیل "free meeting":
  score(spam) = p("free"|spam) × p("meeting"|spam) × p(spam)
              = 0.8 × 0.02 × 0.3 = 0.0048
  score(not-spam) = 0.05 × 0.6 × 0.7 = 0.021
  → not-spam برنده است
```

**چرا "Naïve"؟**
فرض استقلال کلمات واقع‌بینانه نیست ("not good" کاملاً متفاوت از "not" و "good" است) اما در عمل عملکرد خوبی دارد!

**کاربردها:** تشخیص spam، طبقه‌بندی متن، تشخیص بیماری

###  ۴۴–۴۵: ماشین بردار پشتیبان (SVM)



- **مثال دو بُعدی:** نقاط در دو کلاس هستند؛ خطی بیاب که دو کلاس را جدا کند و فاصله از نزدیک‌ترین نقطه در هر کلاس را بیشینه کند (Maximum Margin)
- **n بُعدی:** نقاط توسط یک صفحه جدا می‌شوند
- توابع تبدیل ممکن است غیرخطی باشند و **توابع کرنل (kernel functions)** نامیده می‌شوند
- جداکننده در فضای تبدیل‌شده یک صفحه است، اما در فضای اصلی به منحنی نگاشت می‌شود

#### 📚 توضیح آموزشی

**SVM — ایده اصلی:**

```
کلاس A: ● ● ●
کلاس B: ○ ○ ○
هدف: خطی که بیشترین فاصله از هر دو طرف داشته باشد
                     margin
              ←────────┼────────→
● ● ● ●      |        |        | ○ ○ ○ ○
              ────── hyperplane ──────
```

**Kernel Trick — جادوی SVM:**
وقتی داده خطاً جدا نمی‌شود:
```
2D (خطی نیست) → تبدیل به 3D → در 3D خطاً جدا می‌شود

مثال: x² + y² < 1 (دایره) را نمی‌توانی با خط جدا کنی
اما اگر z = x² + y² بسازی، می‌توانی با صفحه z = 1 جدا کنی!
```

**انواع Kernel:**
- Linear: برای داده‌های خطاً جداپذیر
- RBF (Gaussian): رایج‌ترین، برای داده‌های نامنظم
- Polynomial: برای داده‌های پلینومیال

###  ۴۶–۴۸: شبکه‌های عصبی



- شبکه عصبی لایه‌های متعددی دارد
- لایه اول گره‌های ورودی دارد که از ویژگی‌های ورودی مقداردهی می‌شوند
- هر گره مقادیر ورودی را با یک تابع وزن ترکیب می‌کند
- وزن‌ها با یال‌ها مرتبط هستند
- وزن‌های یال‌ها در مرحله آموزش یاد گرفته می‌شوند
- **Backpropagation:** اگر طبقه‌بندی اشتباه بود، وزن‌ها تنظیم می‌شوند
- **شبکه‌های عصبی عمیق (Deep Neural Networks):** لایه‌های بسیار زیاد
- **یادگیری عمیق (Deep Learning):** آموزش روی نمونه‌های آموزشی بسیار زیاد

#### 📚 توضیح آموزشی

**معماری شبکه عصبی:**

```
ورودی        لایه پنهان ۱    لایه پنهان ۲    خروجی
   ●────────●────────●────────●
   │    ╲  / │    ╲  / │       │
   │     ●   │     ●   │       │
   │    /  ╲ │    /  ╲ │       │
   ●────────●────────●────────●
[input]  [hidden]  [hidden]  [output]
```

**Activation Functions:**
- **Sigmoid:** برای خروجی احتمال (0 تا 1)
- **ReLU:** امروزی‌ترین، سریع‌تر آموزش می‌بیند: `f(x) = max(0, x)`
- **Softmax:** برای multi-class classification (جمع به ۱)

**Backpropagation به زبان ساده:**
```
1. Forward pass: داده را از ابتدا به انتها پیش ببر، نتیجه بگیر
2. Error: خطا = نتیجه واقعی - نتیجه پیش‌بینی‌شده
3. Backward pass: خطا را از انتها به ابتدا پخش کن
4. Update: وزن‌ها را کمی در جهت کاهش خطا تنظیم کن
5. تکرار روی همه نمونه‌ها تا همگرایی
```

**انقلاب Deep Learning:**
- ImageNet 2012: AlexNet → دقت از ۷۵٪ به ۸۵٪ پرید
- امروز: GPT-4, Gemini, Claude — همه deep neural networks هستند
- کاربردها: تشخیص تصویر، ترجمه ماشینی، سنتز گفتار، بازی‌های بردگیر

###  ۴۹: رگرسیون



- **رگرسیون:** پیش‌بینی یک مقدار، نه یک کلاس
- با داشتن مقادیر X₁, X₂, ..., Xₙ می‌خواهیم مقدار Y را پیش‌بینی کنیم
- **رگرسیون خطی:** یافتن ضرایب a₀, a₁, ..., aₙ به طوری که:
  ```
  Y = a₀ + a₁X₁ + a₂X₂ + ... + aₙXₙ
  ```
- **Curve Fitting:** یافتن منحنی‌ای که به داده‌ها برازش دارد
- رگرسیون ضرایبی را می‌یابد که بهترین برازش ممکن را می‌دهند

#### 📚 توضیح آموزشی

**رگرسیون خطی — ساده‌ترین مدل:**

```
y = a + bx
(قیمت خانه) = a + b × (متراژ)

با OLS (Ordinary Least Squares) بهترین a و b را می‌یابیم
که مجموع مربعات خطاها را minimize می‌کند.
```

**انواع رگرسیون:**
| نوع | فرمول | کاربرد |
|-----|-------|---------|
| Linear | y = a + bx | روابط خطی ساده |
| Polynomial | y = a + bx + cx² | روابط منحنی |
| Multiple | y = a + b₁x₁ + b₂x₂ + ... | چند متغیر |
| Logistic | p = 1/(1+e⁻ˣ) | طبقه‌بندی دوتایی |
| Ridge/Lasso | Linear + Regularization | جلوگیری از overfitting |

**Overfitting در رگرسیون:**
```
پلینوم درجه ۱: خط ساده ← underfitting
پلینوم درجه ۱۰: از همه نقاط رد می‌شود ← overfitting
پلینوم درجه ۳-۵: تعادل مناسب ✓
```

###  ۵۰–۵۱: قوانین انجمن



- فروشگاه‌های خرده‌فروش به انجمن‌های بین اقلام مختلفی که مردم می‌خرند علاقه دارند
- کسی که نان می‌خرد احتمالاً شیر هم می‌خرد
- قوانین انجمن: `bread → milk`، `DB-Concepts, OS-Concepts → Networks`
- **پشتیبانی (Support):** چه کسری از جمعیت هم قسمت پیش‌شرط و هم پیامد را دارند
- **اطمینان (Confidence):** چقدر پیامد وقتی پیش‌شرط درست است، درست است

#### 📚 توضیح آموزشی

**Market Basket Analysis — مثال کلاسیک:**

```
تراکنش ۱: {نان، شیر، تخم‌مرغ}
تراکنش ۲: {نان، شیر، کره}
تراکنش ۳: {شیر، تخم‌مرغ، پنیر}
تراکنش ۴: {نان، کره}
تراکنش ۵: {نان، شیر، تخم‌مرغ، کره}

قانون: نان → شیر
Support = تعداد تراکنش‌های حاوی {نان, شیر} / کل تراکنش‌ها
        = 3/5 = 60%

Confidence = تعداد تراکنش‌های حاوی {نان, شیر} / تعداد تراکنش‌های حاوی {نان}
           = 3/4 = 75%
```

**تفسیر:**
- Support بالا: این ترکیب رایج است
- Confidence بالا: اگر نان بخری، احتمال زیادی شیر هم می‌خری

**کاربردهای واقعی:**
- Amazon: "Customers who bought this also bought..."
- Spotify: "چون این آهنگ را دوستی، این را هم گوش بده"
- Netflix: پیشنهاد محتوا
- Walmart: آرایش قفسه‌ها بر اساس آنچه با هم خریده می‌شود

###  ۵۲: خوشه‌بندی



- **خوشه‌بندی:** یافتن خوشه‌هایی از نقاط در داده‌ها به طوری که نقاط مشابه در یک خوشه باشند
- گروه‌بندی نقاط به k مجموعه به طوری که میانگین فاصله نقاط از centroid گروه کمینه باشد
- **Centroid:** نقطه تعریف‌شده با میانگین مختصات در هر بُعد
- **خوشه‌بندی سلسله‌مراتبی:** مثال از طبقه‌بندی زیست‌شناسی

#### 📚 توضیح آموزشی

**K-Means — ساده‌ترین الگوریتم خوشه‌بندی:**

```
مرحله ۱: k نقطه تصادفی به عنوان centroid انتخاب کن
مرحله ۲: هر نقطه را به نزدیک‌ترین centroid اختصاص بده
مرحله ۳: centroid هر خوشه را دوباره محاسبه کن (میانگین)
مرحله ۴: تکرار مرحله ۲ و ۳ تا همگرایی
```

**چگونه k را انتخاب کنیم؟**
- **Elbow Method:** وقتی اضافه کردن خوشه بهبود چندانی نمی‌دهد
- **Silhouette Score:** اندازه‌گیری کیفیت خوشه‌ها

**انواع خوشه‌بندی:**
```
K-Means:    دوایر کروی، حجم مساوی
DBSCAN:     شکل دلخواه، مقاوم به noise
Hierarchical: درخت، نیازی به k نداری
Gaussian Mixture: احتمالاتی، خوشه‌های overlap
```

**کاربرد واقعی:**
- پزشکی: طبقه‌بندی بیماران برای درمان شخصی‌سازی‌شده
- بازاریابی: تقسیم‌بندی مشتریان
- تشخیص ناهنجاری: نقاطی که در هیچ خوشه‌ای جا نمی‌گیرند

###  ۵۳: خوشه‌بندی و Collaborative Filtering



**هدف:** پیش‌بینی فیلم/کتاب/... که ممکن است فردی به آن علاقه داشته باشد

**رویکرد مبتنی بر خوشه‌بندی تکراری:**

3. دوباره خوشه‌بندی افراد
4. تکرار تا تعادل

#### 📚 توضیح آموزشی

**Collaborative Filtering — قلب سیستم‌های توصیه:**

```
User-Based CF:
"کاربرانی مثل تو فیلم X را دوستند → شاید تو هم دوستی بداری"

Item-Based CF:
"فیلم‌هایی که با فیلم X دوست‌داشته می‌شوند → Y و Z نیز"

Matrix Factorization (رویکرد مدرن):
ماتریس rating‌ها را به دو ماتریس کوچک تجزیه کن:
User_Matrix × Item_Matrix ≈ Rating_Matrix
```

**مشکلات معروف:**
- **Cold Start:** کاربر/محصول جدید — هیچ تاریخچه‌ای نداری

- **Filter Bubble:** فقط چیزهایی پیشنهاد می‌دهی که قبلاً دوست داشته

###  ۵۶–۵۸: بهترین تقسیم‌بندی‌ها در درخت تصمیم



**معیار Gini:**
```
Gini(S) = 1 - Σpᵢ²
```
وقتی همه نمونه‌ها در یک کلاس: Gini = 0 (خالص)

**معیار Entropy:**
```
entropy(S) = -Σ pᵢ log₂(pᵢ)
```

**Information Gain:**
```
Information-gain(S, {S₁,...,Sᵣ}) = purity(S) - purity(S₁,...,Sᵣ)
```

**نسبت Information Gain:**
```
Gain Ratio = Information-gain / Information-content
```
بهترین تقسیم: بیشترین نسبت information gain

#### 📚 توضیح آموزشی

**چرا به معیار خلوص نیاز داریم؟**

تصور کن می‌خواهی داده‌ای با ۵۰٪ spam و ۵۰٪ non-spam را تقسیم کنی:

```
تقسیم خوب:
  شاخه ۱: 90% spam, 10% non-spam   → خلوص بالا
  شاخه ۲: 10% spam, 90% non-spam   → خلوص بالا

تقسیم بد:
  شاخه ۱: 50% spam, 50% non-spam   → خلوص پایین (مثل قبل!)
  شاخه ۲: 50% spam, 50% non-spam
```

**مقایسه Gini و Entropy:**
- هر دو نتایج مشابه می‌دهند
- Gini محاسبه سریع‌تری دارد (بدون log)
- Decision Tree در sklearn از Gini استفاده می‌کند
- C4.5 از Entropy استفاده می‌کند

**Gain Ratio چرا مهم است؟**
Information Gain ویژگی‌هایی با ارزش‌های زیاد را ترجیح می‌دهد (مثل ID). Gain Ratio این bias را تصحیح می‌کند.

###  ۵۹–۶۰: الگوریتم ساخت درخت تصمیم



**الگوریتم بازگشتی:**
```
Procedure GrowTree(S):
  Partition(S)

Procedure Partition(S):
  if purity(S) > p OR |S| < s:
    return  // برگ شدیم
  for each attribute A:
    evaluate splits on A
  Use best split to partition S into S₁, ..., Sᵣ
  for i = 1 to r:
    Partition(Sᵢ)
```

#### 📚 توضیح آموزشی

**ID3, C4.5, CART — الگوریتم‌های اصلی:**

| الگوریتم | معیار | ویژگی |
|----------|-------|-------|
| ID3 | Information Gain | فقط categorical |
| C4.5 | Gain Ratio | categorical + continuous |
| CART | Gini | binary tree، regression هم |

**Pre-pruning vs Post-pruning:**
```
Pre-pruning: در حین ساخت درخت توقف کن
  → شرط: اگر purity > p یا |S| < s

Post-pruning: درخت کامل بساز، سپس هرس کن
  → با Validation Set تصمیم بگیر کدام شاخه‌ها ارزش ندارند
```

**Ensemble Methods (بهتر از یک درخت!):**
```
Random Forest: ۱۰۰ درخت با feature و sample random
  → رأی‌گیری: کلاسی که بیشترین رأی را دارد
  
Gradient Boosting (XGBoost, LightGBM):
  → هر درخت خطای درخت قبلی را جبران می‌کند
  → امروز قوی‌ترین روش برای Tabular Data
```

###  ۶۱–۶۲: یافتن قوانین انجمن



**الگوریتم ساده:**
1. همه مجموعه‌های ممکن از اقلام را در نظر بگیر
2. پشتیبانی هر مجموعه را بیاب
3. **Itemset های بزرگ:** مجموعه‌هایی با پشتیبانی کافی
4. از itemset های بزرگ قوانین انجمن تولید کن

**تکنیک Apriori:**
- Pass 1: همه مجموعه‌های ۱ آیتمی → حذف کم‌پشتیبانی
- Pass i: کاندیداها: هر مجموعه i آیتمی که همه زیرمجموعه‌های i-1 آیتمی آن بزرگ هستند

#### 📚 توضیح آموزشی

**اصل Apriori — خلاصه‌سازی فضای جستجو:**

```
اگر {نان, شیر} کم‌پشتیبانی است →
{نان, شیر, تخم‌مرغ} هم کم‌پشتیبانی است!
(زیرمجموعه هر مجموعه کم‌پشتیبانی، کم‌پشتیبانی است)
```

این "قانون ضدیکنواختی" (Anti-monotonicity) امکان Pruning عظیم فضای جستجو را می‌دهد.

**FP-Growth — الگوریتم سریع‌تر:**
Apriori برای مجموعه‌های بزرگ کند است (passes زیاد روی داده). FP-Growth:
1. داده را فشرده در یک درخت (FP-Tree) ذخیره می‌کند
2. بدون generation صریح کاندیداها، itemset های بزرگ را استخراج می‌کند
3. ۱۰-۱۰۰× سریع‌تر از Apriori

###  ۶۳: سایر انواع انجمن



- **انحراف از احتمال مورد انتظار** جالب‌تر است
- علاقه‌مندی به همبستگی مثبت و منفی
  - **همبستگی مثبت:** هم‌رخداد بیشتر از پیش‌بینی
  - **همبستگی منفی:** هم‌رخداد کمتر از پیش‌بینی
- **انجمن‌های توالی:** مثلاً هر بار که اوراق قرضه بالا می‌رود، قیمت سهام ۲ روز بعد پایین می‌آید
- **انحراف از الگوهای زمانی**

#### 📚 توضیح آموزشی

**Lift — معیار بهتر از Confidence:**

```
Lift(نان → شیر) = Confidence(نان → شیر) / P(شیر)

اگر Lift = 1: نان و شیر مستقل هستند (قانون بی‌ارزش)
اگر Lift > 1: همبستگی مثبت (جالب!)
اگر Lift < 1: همبستگی منفی (هم جالب!)
```

**Sequential Pattern Mining:**
```
مثال فروشگاه آنلاین:
کاربر A: [iPhone] → [AirPods] → [Apple Watch]
کاربر B: [iPhone] → [AirPods] → [MagSafe Charger]
کاربر C: [iPhone] → [case] → [AirPods]

الگو: iPhone → AirPods (در ۸۰٪ موارد)
→ بعد از خرید iPhone، AirPods پیشنهاد بده
```

###  ۶۴–۶۵: خوشه‌بندی سلسله‌مراتبی و الگوریتم BIRCH



**الگوریتم‌های Agglomerative:**
- خوشه‌های کوچک بساز، سپس آن‌ها را در خوشه‌های بزرگ‌تر ترکیب کن

**الگوریتم‌های Divisive:**
- با همه اقلام در یک خوشه شروع کن، به تدریج خوشه‌ها را تقسیم کن

**الگوریتم BIRCH:**
- از R-tree در حافظه برای ذخیره نقاط استفاده می‌کند
- نقاط را یک به یک درج کن
- اگر نود جدید به خوشه‌ای نزدیک است، با آن ادغام کن
- در انتها تعداد زیادی خوشه در برگ‌های R-tree

#### 📚 توضیح آموزشی

**Hierarchical Clustering — Dendrogram:**

```
              ┌──────────────────────┐
              │                      │
         ┌───┴───┐              ┌───┴───┐
         │       │              │       │
     ┌───┴───┐   C          ┌──┴──┐    E
     │       │              │     │
     A       B              D     (...)

→ می‌توانی با "قطع" در هر ارتفاع، k دلخواه خوشه داشته باشی
```

**BIRCH — برای داده‌های بزرگ:**

مشکل K-Means: باید همه داده را در حافظه داشته باشی.
BIRCH این مشکل را با درخت فشرده حل می‌کند:
```
CF (Clustering Feature): هر خوشه با (N, LS, SS) خلاصه می‌شود
  N = تعداد نقاط
  LS = مجموع خطی
  SS = مجموع مربع
→ بدون نیاز به ذخیره تک‌تک نقاط!
```
