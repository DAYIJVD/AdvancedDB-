##  ۳ — داده‌های نیمه‌ساختاریافته


**داده‌های نیمه‌ساختاریافته**

- بسیاری از کاربردها نیاز به ذخیره‌سازی داده‌های پیچیده دارند که طرح‌واره‌شان مکرراً تغییر می‌کند
- الزام مدل رابطه‌ای به انواع داده اتمی ممکن است بیش از حد سختگیرانه باشد
  - مثال: ذخیره مجموعه‌ای از علایق به‌عنوان یک ویژگی چندمقداری در پروفایل کاربر، ساده‌تر از نرمال‌سازی آن است
- تبادل داده می‌تواند از داده‌های نیمه‌ساختاریافته بهره فراوانی ببرد
  - تبادل می‌تواند بین برنامه‌ها، یا بین لایه پشتی و رابط کاربری باشد
  - وب‌سرویس‌ها امروزه به‌طور گسترده استفاده می‌شوند، با داده‌های پیچیده‌ای که به رابط کاربری ارسال و توسط اپلیکیشن موبایل یا JavaScript نمایش داده می‌شوند
- JSON و XML مدل‌های داده نیمه‌ساختاریافته پرکاربردی هستند

### توضیح آموزشی:
**داده نیمه‌ساختاریافته (Semi-Structured Data)** داده‌ای است که طرح‌واره ثابت ندارد اما دارای ساختار درونی است. این مفهوم در تضاد با دو نوع دیگر است:

| نوع داده | مثال | ویژگی |
|---|---|---|
| ساختاریافته | جدول SQL | طرح‌واره ثابت، همه ستون‌ها مشخص |
| نیمه‌ساختاریافته | JSON، XML | طرح‌واره انعطاف‌پذیر |
| بدون ساختار | متن آزاد، تصویر | فاقد طرح‌واره |

**چرا مهم است؟** در دنیای واقعی، داده‌های کاربران اغلب یکسان نیستند. یک کاربر ممکن است ۳ علاقه داشته باشد و دیگری ۲۰ تا. نگه‌داشتن این‌ها در جداول رابطه‌ای نیازمند Join های پیچیده است، در حالی که یک آرایه JSON مشکل را ساده حل می‌کند.

##  ۴ — ویژگی‌های مدل‌های داده نیمه‌ساختاریافته (بخش اول)


**ویژگی‌های مدل‌های داده نیمه‌ساختاریافته**

- **طرح‌واره انعطاف‌پذیر**
  - نمایش ستون گسترده: به هر تاپل اجازه داده می‌شود مجموعه متفاوتی از ویژگی‌ها داشته باشد؛ می‌توان در هر زمان ویژگی جدید اضافه کرد
  - نمایش ستون پراکنده: طرح‌واره دارای مجموعه ثابت اما بزرگی از ویژگی‌هاست، اما هر تاپل ممکن است تنها زیرمجموعه‌ای را ذخیره کند
- **انواع داده چندمقداری**
  - مجموعه‌ها (Sets) و مجموعه‌های مرتب (Multisets)
    - مثال: مجموعه علایق: {'بسکتبال'، 'لالیگا'، 'آشپزی'، 'انیمه'، 'جاز'}
  - نگاشت کلید-مقدار (Key-Value Map)
    - ذخیره مجموعه‌ای از جفت‌های کلید-مقدار
    - مثال: {(برند، Apple)، (ID، MacBook Air)، (اندازه، ۱۳)، (رنگ، نقره‌ای)}
    - عملیات: put(key, value)، get(key)، delete(key)
  - آرایه‌ها (Arrays)
    - به‌طور گسترده برای کاربردهای علمی و پایش استفاده می‌شوند

### توضیح آموزشی:
این  سه مکانیزم اصلی انعطاف‌پذیری در مدل‌های نیمه‌ساختاریافته را معرفی می‌کند:

**۱. طرح‌واره انعطاف‌پذیر:**
- **Wide Column**: پایگاه داده‌هایی مثل Apache Cassandra و HBase از این الگو استفاده می‌کنند. هر سطر می‌تواند ستون‌های کاملاً متفاوتی داشته باشد.
- **Sparse Column**: پایگاه داده‌هایی مثل Google Bigtable. همه ستون‌ها از قبل تعریف شده‌اند اما لازم نیست پر باشند.

**۲. نگاشت کلید-مقدار:** این ساختار پایه Redis، DynamoDB و بسیاری از سیستم‌های NoSQL است. ساده اما قدرتمند.

**۳. آرایه‌ها:** برای داده‌های سری زمانی (مثل سنسورهای IoT) بسیار کارآمدند.

##  ۵ — ویژگی‌های مدل‌های داده نیمه‌ساختاریافته (بخش دوم)


**ویژگی‌های مدل‌های داده نیمه‌ساختاریافته (ادامه)**

- **آرایه‌ها**
  - به‌طور گسترده برای کاربردهای علمی و پایشی استفاده می‌شوند
  - مثال: قرائت‌های انجام‌شده در فواصل منظم می‌توانند به‌جای جفت‌های (زمان، مقدار) به‌صورت آرایه‌ای از مقادیر نمایش داده شوند
  - `[5, 8, 9, 11]` به‌جای `{(1,5), (2,8), (3,9), (4,11)}`
- **انواع ویژگی‌های چندمقداری**
  - با مدل داده فرم غیر اول نرمال (NFNF) مدل‌سازی می‌شوند
  - امروزه توسط اکثر سیستم‌های پایگاه داده پشتیبانی می‌شوند
- **پایگاه داده آرایه**: پایگاه داده‌ای که پشتیبانی تخصصی از آرایه‌ها ارائه می‌دهد
  - مانند: ذخیره‌سازی فشرده، توسعه‌های زبان پرس‌وجو و غیره
  - Oracle GeoRaster، PostGIS، SciDB و غیره

### توضیح آموزشی:
**فرم اول نرمال (1NF)** در مدل رابطه‌ای اصل پایه‌ای است که می‌گوید هر خانه جدول باید یک مقدار اتمی (تقسیم‌ناپذیر) داشته باشد. اما در **NFNF (Non First Normal Form)**، یک خانه می‌تواند خودش یک مجموعه یا آرایه باشد.

**مقایسه رویکردها:**
```
رویکرد رابطه‌ای (1NF):
جدول readings: (sensor_id, timestamp, value)
(s1, 10:00, 5), (s1, 10:01, 8), (s1, 10:02, 9) ...

رویکرد NFNF:
جدول sensors: (sensor_id, readings[])
(s1, [5, 8, 9, 11])
```
رویکرد دوم حجم ذخیره‌سازی را کاهش می‌دهد و پرس‌وجوهای روی سری‌های زمانی را سریع‌تر می‌کند.

**پایگاه داده‌های آرایه‌ای** مانند SciDB برای علوم داده و هوش مصنوعی که نیاز به ماتریس‌های بزرگ دارند بسیار مناسب هستند.

##  ۶ — انواع داده‌ی تودرتو


**انواع داده‌ی تودرتو**

- داده‌های سلسله‌مراتبی در بسیاری از کاربردها رایج هستند
- **JSON: نشانه‌گذاری شیء JavaScript**
  - امروزه به‌طور گسترده استفاده می‌شود
- **XML: زبان نشانه‌گذاری توسعه‌پذیر**
  - نشانه‌گذاری نسل قبل، هنوز هم به‌طور گسترده استفاده می‌شود

### توضیح آموزشی:
**داده‌های سلسله‌مراتبی** در طبیعت فراوانند: یک سفارش خرید شامل چندین قلم است، هر قلم مشخصاتی دارد، هر مشخصات زیرمشخصاتی. مدل رابطه‌ای این داده‌ها را در چندین جدول توزیع می‌کند، اما JSON و XML آن‌ها را به‌صورت تودرتو در یک سند نگه می‌دارند.

**مقایسه JSON و XML:**
| ویژگی | JSON | XML |
|---|---|---|
| خوانایی | بهتر برای کد | بهتر برای انسان |
| حجم | کمتر | بیشتر (verbose) |
| کاربرد امروز | API های REST | پروتکل‌های قدیمی‌تر، SOAPَ |
| پشتیبانی پایگاه داده | گسترده | خوب |

##  ۷ — JSON


**JSON**

- نمایش متنی که به‌طور گسترده برای تبادل داده استفاده می‌شود
- مثال داده JSON:
```json
{
    "ID": "22222",
    "name": {
        "firstname": "Albert",
        "lastname": "Einstein"
    },
    "deptname": "Physics",
    "children": [
        {"firstname": "Hans", "lastname": "Einstein"},
        {"firstname": "Eduard", "lastname": "Einstein"}
    ]
}
```
- انواع: عدد صحیح، اعشاری، رشته، و بولین
- **اشیاء**: نگاشت کلید-مقدار، یعنی مجموعه‌ای از جفت‌های (نام ویژگی، مقدار)
- **آرایه‌ها** نیز نگاشت کلید-مقدار هستند (از اندیس به مقدار)

### توضیح آموزشی:
**ساختار JSON** چهار مفهوم اصلی دارد:

۱. **اشیاء (`{}`):** مجموعه کلید-مقدار، مثل یک رکورد پایگاه داده
۲. **آرایه‌ها (`[]`):** لیست مرتب از مقادیر، ایندکس‌گذاری از صفر
۳. **مقادیر اتمی:** رشته، عدد، boolean، null
۴. **تودرتویی:** اشیاء و آرایه‌ها می‌توانند یکدیگر را شامل شوند

**در مثال بالا:**
- `name` خودش یک شیء JSON است (تودرتویی)
- `children` یک آرایه از اشیاء است
- این ساختار نشان‌دهنده رابطه‌های یک‌به‌یک (`name`) و یک‌به‌چند (`children`) است

**نکته فنی مهم:** JSON آرایه‌ها را به‌عنوان نگاشت از اندیس (0, 1, 2, ...) به مقدار می‌بیند — این همان مفهوم key-value map است.

##  ۸ — JSON (ادامه)


**JSON (ادامه)**

- JSON امروزه در تبادل داده فراگیر است
  - به‌طور گسترده برای وب‌سرویس‌ها استفاده می‌شود
  - اکثر برنامه‌های مدرن حول وب‌سرویس‌ها طراحی شده‌اند
- توسعه‌های SQL برای JSON:
  - انواع JSON برای ذخیره‌سازی داده JSON
  - استخراج داده از اشیاء JSON با عبارات مسیر (Path Expressions)
    - مثال: `V->ID` یا `v.ID`
  - تولید JSON از داده رابطه‌ای
    - مثال: `json_build_object('ID', 12345, 'name', 'Einstein')`
  - ایجاد مجموعه‌های JSON با تجمیع
    - مثال: تابع تجمیعی `json_agg` در PostgreSQL
  - نحو دستوری در پایگاه داده‌های مختلف متفاوت است
- JSON پرحجم است
  - نمایش‌های فشرده مانند BSON (Binary JSON) برای ذخیره‌سازی کارآمد داده استفاده می‌شوند

### توضیح آموزشی:
**پشتیبانی SQL از JSON** در پایگاه داده‌های مدرن بسیار قوی شده است. بیایید با مثال‌های عملی ببینیم:

```sql
-- ذخیره JSON در PostgreSQL
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    profile JSONB  -- JSONB بهتر از JSON است (ذخیره باینری)
);

-- استخراج فیلد
SELECT profile->'name'->>'firstname' FROM users WHERE id = 1;

-- ساخت JSON از داده رابطه‌ای
SELECT json_build_object('ID', id, 'name', name) FROM instructors;

-- تجمیع چندین سطر به آرایه JSON
SELECT json_agg(name) FROM students WHERE dept = 'CS';
```

**BSON vs JSON:**
- JSON: متن خوانا، حجم بیشتر
- BSON: باینری، حجم کمتر، عملیات سریع‌تر — پایگاه داده MongoDB از آن استفاده می‌کند

##  ۹ — XML


**XML**

- XML از تگ‌ها برای علامت‌گذاری متن استفاده می‌کند
```xml
<course>
    <course_id> CS-101 </course_id>
    <title> مقدمه‌ای بر علوم کامپیوتر </title>
    <dept_name> Comp. Sci. </dept_name>
    <credits> 4 </credits>
</course>
```
- تگ‌ها داده را خودتوصیف (self-documenting) می‌کنند
- تگ‌ها می‌توانند سلسله‌مراتبی باشند

### توضیح آموزشی:
**XML (Extensible Markup Language)** در سال ۱۹۹۸ توسط W3C معرفی شد و برای مدت‌ها استاندارد تبادل داده بود.

**ویژگی‌های کلیدی XML:**
- **خودتوصیف بودن:** نام تگ معنای محتوا را توضیح می‌دهد (`<credits>4</credits>` بدون مستندات قابل فهم است)
- **سلسله‌مراتبی:** تگ‌ها می‌توانند تودرتو باشند، درخت DOM ایجاد می‌کنند
- **توسعه‌پذیر:** می‌توانید هر نام تگی بسازید (برخلاف HTML که تگ‌های ثابت دارد)

**مقایسه ساختاری:**
```xml
<!-- XML - پرحجم‌تر اما صریح‌تر -->
<person><name>Ali</name><age>25</age></person>

// JSON - فشرده‌تر
{"name": "Ali", "age": 25}
```

##  ۱۰ — نمونه داده در XML


**نمونه داده در XML**
```xml
<purchase_order>
    <identifier> P-101 </identifier>
    <purchaser>
        <name> کریزی کایوت </name>
        <address> جاده ۶۶، مسا فلتس، آریزونا، آمریکا </address>
    </purchaser>
    <supplier>
        <name> Acme Supplies </name>
        <address> ۱ برادوی، نیویورک، آمریکا </address>
    </supplier>
    <itemlist>
        <item>
            <identifier> RS1 </identifier>
            <description> سورتمه موشک‌پران اتمی </description>
            <quantity> 2 </quantity>
            <price> 199.95 </price>
        </item>
        <item>…</item>
    </itemlist>
    <total_cost> 429.85 </total_cost>
</purchase_order>
```

### توضیح آموزشی:
این مثال یک **سند سفارش خرید (Purchase Order)** واقعی را نشان می‌دهد که در تجارت الکترونیک و سیستم‌های ERP استفاده می‌شود.

**چرا XML برای این مثال خوب است؟**
- ساختار سفارش خرید سلسله‌مراتبی است: سفارش → آیتم‌ها → جزئیات
- هر سفارش می‌تواند تعداد متفاوتی آیتم داشته باشد (رابطه یک‌به‌چند)
- تگ‌ها به‌قدری گویا هستند که بدون مستندات قابل درک هستند

**در مدل رابطه‌ای این داده نیاز به ۳ جدول داشت:**
```
Orders(id, purchaser_id, supplier_id, total_cost)
OrderItems(order_id, item_id, quantity, price)
Parties(id, name, address)
```
XML همه این‌ها را در یک سند نگه می‌دارد.

##  ۱۱ — XML (ادامه)


**XML (ادامه)**

- زبان XQuery برای پرس‌وجوی ساختارهای XML تودرتو توسعه یافته است
  - در حال حاضر به‌طور گسترده استفاده نمی‌شود
- توسعه‌های SQL برای پشتیبانی از XML:
  - ذخیره‌سازی داده XML
  - تولید داده XML از داده رابطه‌ای
  - استخراج داده از انواع داده XML
  - عبارات مسیر (Path Expressions)
- برای اطلاعات بیشتر به فصل ۳۰ (آنلاین) مراجعه کنید

### توضیح آموزشی:
**XQuery** زبانی مانند SQL اما برای XML است. مثال:
```xquery
(* پیدا کردن تمام درس‌های ۴ واحدی *)
for $c in //course
where $c/credits = 4
return $c/title
```

**اما چرا XQuery محبوب نشد؟**
- پیچیده‌تر از SQL بود
- JSON و REST API ها جای XML و SOAP را گرفتند
- پایگاه داده‌های NoSQL با پشتیبانی بومی از JSON ظهور کردند

**امروزه** اکثر پایگاه داده‌های رابطه‌ای (PostgreSQL, Oracle, SQL Server) انواع داده XML دارند و از XPath/XQuery برای پرس‌وجو پشتیبانی می‌کنند، اما استفاده از JSON بسیار رایج‌تر است.

##  ۱۲ — نمایش دانش


**نمایش دانش**

- نمایش دانش انسانی یک هدف دیرینه هوش مصنوعی است
- در طول زمان، نمایش‌های مختلفی از واقعیت‌ها و قواعد استنتاج پیشنهاد شده‌اند
- **RDF: فرمت توصیف منبع**
  - نمایش ساده‌شده برای واقعیت‌ها، به‌صورت سه‌گانه (Triple)
  - (موضوع، محمول، مفعول) — (subject, predicate, object)
  - مثال: (NBA-2019، قهرمان، Raptors)
  - مثال: (واشنگتن دی‌سی، پایتخت، آمریکا)
  - مثال: (واشنگتن دی‌سی، جمعیت، ۶,۲۰۰,۰۰۰)
- اشیاء دارای ویژگی‌ها و روابط با اشیاء دیگر را مدل‌سازی می‌کند
- مانند مدل ER اما با طرح‌واره انعطاف‌پذیر
  - (ID، نام_ویژگی، مقدار)
  - (ID1، نام_رابطه، ID2)
- دارای نمایش گراف طبیعی است

### توضیح آموزشی:
**RDF (Resource Description Framework)** پایه معنایی وب (Semantic Web) است که W3C آن را معرفی کرد.

**مدل سه‌گانه RDF:**
```
موضوع (Subject) → محمول (Predicate) → مفعول (Object)
     ↓                    ↓                   ↓
"علی رضایی"        "تدریس_می‌کند"         "پایگاه داده"
"پایگاه داده"      "دپارتمان"             "علوم کامپیوتر"
```

**گراف دانش (Knowledge Graph) چیست؟**
همین سه‌گانه‌ها اگر در قالب گراف رسم شوند، هر موضوع و مفعول یک گره و هر محمول یک یال جهت‌دار است. گوگل از Knowledge Graph برای نمایش اطلاعات در کنار نتایج جستجو استفاده می‌کند.

**تفاوت با ER:**
- در ER، طرح‌واره از قبل ثابت است
- در RDF، می‌توان در هر لحظه رابطه‌ی جدیدی اضافه کرد

##  ۱۳ — نمای گراف از داده RDF


**نمای گراف از داده RDF**

گراف دانش (Knowledge Graph)

### توضیح آموزشی:
**گراف دانش** مجموعه‌ای از سه‌گانه‌های RDF است که به‌صورت بصری نمایش داده می‌شود:
- **گره‌ها (Nodes):** موجودیت‌ها (افراد، مکان‌ها، مفاهیم)
- **یال‌ها (Edges):** روابط بین موجودیت‌ها

- **Freebase:** خریداری‌شده توسط گوگل (اکنون در Wikidata ادغام شده)

گراف‌های دانش پایه سیستم‌های پاسخ به سوال (QA) و دستیارهای هوش مصنوعی مانند Siri و Alexa هستند.

##  ۱۵ — پرس‌وجوی RDF: SPARQL


**پرس‌وجوی RDF: SPARQL**

- الگوهای سه‌گانه:
  - `?cid title "مقدمه‌ای بر علوم کامپیوتر"`
  - `?sid course ?cid`
- پرس‌وجوهای SPARQL:
```sparql
SELECT ?name
WHERE {
    ?cid title "Intro. to Computer Science" .
    ?sid course ?cid .
    ?id takes ?sid .
    ?id name ?name .
}
```
- همچنین پشتیبانی از:
  - تجمیع، اتصال اختیاری (مشابه outer join)، زیرپرس‌وجو و غیره
  - بستار انتقالی روی مسیرها

### توضیح آموزشی:
**SPARQL** (تلفظ: اسپارکل) زبان پرس‌وجو برای RDF است، مانند SQL برای پایگاه داده رابطه‌ای.

**تشبیه با SQL:**
```sql
-- SQL: پیدا کردن نام دانشجویانی که درس CS-101 را می‌گیرند
SELECT s.name
FROM students s
JOIN takes t ON s.id = t.student_id
JOIN sections sec ON t.section_id = sec.id
JOIN courses c ON sec.course_id = c.id
WHERE c.title = 'Intro. to Computer Science';

-- SPARQL: همان پرس‌وجو
SELECT ?name WHERE {
    ?cid title "Intro. to Computer Science" .
    ?sid course ?cid .
    ?id takes ?sid .
    ?id name ?name .
}
```

**علامت سوال (`?`)** نشان‌دهنده متغیر است که باید پیدا شود.
**نقطه (`.`)** پایان یک الگوی سه‌گانه را نشان می‌دهد.

**قابلیت‌های پیشرفته SPARQL:**
- `OPTIONAL {}`: مشابه LEFT OUTER JOIN
- `FILTER`: فیلتر کردن نتایج
- `UNION`: ترکیب نتایج
- Property Paths: دنبال کردن مسیرهای دلخواه در گراف

##  ۱۶ — نمایش RDF (ادامه)


**نمایش RDF (ادامه)**

- سه‌گانه‌های RDF روابط دوتایی را نمایش می‌دهند
- چگونه روابط n-تایی را نمایش دهیم؟
- **رویکرد ۱:** ایجاد موجودیت مصنوعی و اتصال به هر یک از n موجودیت
  - مثال: (باراک اوباما، رئیس‌جمهور، آمریکا، ۲۰۰۸-۲۰۱۶) را می‌توان چنین نمایش داد:
  - `(e1, person, Barack Obama), (e1, country, USA),`
  - `(e1, president-from, 2008), (e1, president-till, 2016)`
- **رویکرد ۲:** استفاده از چهارگانه (Quads) به‌جای سه‌گانه، با موجودیت زمینه
  - مثال: `(Barack Obama, president-of, USA, c1)`
  - `(c1, president-from, 2008), (c1, president-till, 2016)`
- RDF به‌طور گسترده به‌عنوان نمایش پایگاه دانش استفاده می‌شود
  - DBpedia، Yago، Freebase، WikiData و غیره
- پروژه Linked Open Data هدف اتصال گراف‌های دانش مختلف را دارد تا پرس‌وجوها بتوانند پایگاه داده‌ها را در بر بگیرند

### توضیح آموزشی:
**مشکل روابط n-تایی در RDF:**
RDF ذاتاً برای روابط **دوتایی** (binary) طراحی شده. اما واقعیت‌های دنیای واقعی اغلب بیش از دو موجودیت دارند.

**مثال:** رئیس‌جمهوری اوباما ۴ بعد دارد: شخص، کشور، تاریخ شروع، تاریخ پایان

**رویکرد ۱ - Reification (تشیی‌سازی):**
```
e1 → person → "Barack Obama"
e1 → country → USA
e1 → from → 2008
e1 → till → 2016
```
یک گره مصنوعی `e1` ایجاد می‌کنیم که همه ابعاد به آن وصل هستند.

**رویکرد ۲ - Named Graphs (گراف‌های نام‌گذاری‌شده):**
به‌جای سه‌گانه، از چهارگانه `(S, P, O, Context)` استفاده می‌کنیم. این رویکرد پیشرفته‌تر است و در پایگاه داده‌هایی مثل Wikidata مورد استفاده قرار می‌گیرد.

**Linked Open Data** یکی از بزرگترین پروژه‌های وب معنایی است که هزاران پایگاه دانش را با استانداردهای RDF به هم متصل می‌کند.

##  ۱۷ — شیءگرایی


**شیءگرایی**

- مدل داده شیء-رابطه‌ای سیستم نوعی غنی‌تری با انواع داده پیچیده و شیءگرایی فراهم می‌کند
- برنامه‌ها اغلب به زبان‌های برنامه‌نویسی شیءگرا نوشته می‌شوند
  - سیستم نوع با سیستم نوع رابطه‌ای مطابقت ندارد
  - جابجایی بین زبان دستوری و SQL دشوار است
- رویکردهای ادغام شیءگرایی با پایگاه داده:
  - ساخت پایگاه داده شیء-رابطه‌ای، با افزودن قابلیت‌های شیءگرا به پایگاه داده رابطه‌ای
  - تبدیل خودکار داده بین مدل زبان برنامه‌نویسی و مدل رابطه‌ای؛ تبدیل داده با Object-Relational Mapping مشخص می‌شود
  - ساخت پایگاه داده شیءگرا که به‌صورت بومی از داده شیءگرا پشتیبانی کرده و دسترسی مستقیم از زبان برنامه‌نویسی را ممکن می‌سازد

### توضیح آموزشی:
**تناقض عدم تطابق امپدانسی (Impedance Mismatch):**
```java
// در Java داریم:
class Student {
    String name;
    List<Course> courses;  // لیست درس‌ها
    Address address;       // آدرس به‌عنوان شیء
}

// اما در SQL:
-- Student(id, name, street, city)  ← address نمی‌تواند شیء باشد
-- StudentCourses(student_id, course_id)  ← courses نمی‌تواند لیست باشد
```

این عدم تطابق **Impedance Mismatch** نامیده می‌شود و یکی از بزرگترین چالش‌های توسعه‌دهندگان است.

**سه راه‌حل:**
1. **Object-Relational DB:** مثل PostgreSQL که از انواع تعریف‌شده توسط کاربر پشتیبانی می‌کند
2. **ORM:** مثل Hibernate (Java) و Django ORM (Python) — محبوب‌ترین راه‌حل
3. **Object-Oriented DB:** مثل db4o — کمتر رایج

**سیستم‌های پایگاه داده شیء-رابطه‌ای**

- انواع تعریف‌شده توسط کاربر:

    ID varchar(20) primary key,

- نحو دستوری در پایگاه داده‌های مختلف متفاوت است

### توضیح آموزشی:
**پایگاه داده‌های شیء-رابطه‌ای (ORDB)** بهترین هر دو دنیا را ترکیب می‌کنند:

**مزایا:**
- قدرت SQL و ACID برای تراکنش‌ها

##  ۱۹ — ارث‌بری نوع و جدول


**ارث‌بری نوع و جدول**

- ارث‌بری نوع:
```sql
CREATE TYPE Student UNDER Person (
    degree varchar(20)
);
CREATE TYPE Teacher UNDER Person (
    salary integer
);
```
- نحو ارث‌بری جدول در PostgreSQL و Oracle:
```sql
-- PostgreSQL
CREATE TABLE students (degree varchar(20))
    INHERITS people;
CREATE TABLE teachers (salary integer)
    INHERITS people;

-- Oracle
CREATE TABLE people OF Person;
CREATE TABLE students OF Student UNDER people;
CREATE TABLE teachers OF Teacher UNDER people;
```

### توضیح آموزشی:
**ارث‌بری در پایگاه داده** مفهوم OOP را به دنیای SQL می‌آورد.

**سلسله‌مراتب:**
```
        Person
       /       \
  Student    Teacher
```

**در PostgreSQL با `INHERITS`:**
```sql
-- وقتی از students پرس‌وجو می‌کنید، تمام ستون‌های people هم هستند
SELECT * FROM students;
-- ID، name، address (از Person) + degree (از Student)

-- پرس‌وجو از کل سلسله‌مراتب:
SELECT * FROM ONLY people;  -- فقط people
SELECT * FROM people;        -- people + students + teachers
```

**پلی‌مورفیسم در پرس‌وجو:**
```sql
-- همه افراد (دانشجویان + اساتید + بقیه) را پیدا کن
SELECT name FROM people WHERE city = 'Tehran';
```
این از قدرتمندترین ویژگی‌های ORDB است.

##  ۲۰ — انواع مرجع


**انواع مرجع**

- ایجاد انواع مرجع:
```sql
CREATE TYPE Person (
    ID varchar(20) primary key,
    name varchar(20),
    address varchar(20)
) ref from(ID);

CREATE TABLE people OF Person;

CREATE TYPE Department (
    dept_name varchar(20),
    head ref(Person) SCOPE people
);

CREATE TABLE departments OF Department;
INSERT INTO departments VALUES ('CS', '12345');
```
- مراجع تولیدشده توسط سیستم را می‌توان با زیرپرس‌وجو بازیابی کرد:
```sql
(SELECT ref(p) FROM people AS p WHERE ID = '12345')
```
- استفاده از مراجع در عبارات مسیر:
```sql
SELECT head->name, head->address FROM departments;
```

### توضیح آموزشی:
**نوع مرجع (Reference Type)** معادل پوینتر در زبان‌های برنامه‌نویسی است، اما با integrity constraint های پایگاه داده.

**تفاوت با کلید خارجی:**
| ویژگی | کلید خارجی (FK) | مرجع (ref) |
|---|---|---|
| نحو دسترسی | نیاز به JOIN | مسیر مستقیم (`->`) |
| مدیریت | دستی | نیمه‌خودکار |
| OOP-like | خیر | بله |

**مزیت اصلی:**
```sql
-- با FK نیاز به JOIN داریم:
SELECT p.name, p.address
FROM departments d JOIN people p ON d.head_id = p.ID
WHERE d.dept_name = 'CS';

-- با مرجع، مستقیم:
SELECT head->name, head->address
FROM departments
WHERE dept_name = 'CS';
```
این خوانایی کد را بسیار بهتر می‌کند، به‌خصوص وقتی چندین سطح تودرتویی داریم.

##  ۲۱ — نگاشت شیء-رابطه‌ای


**نگاشت شیء-رابطه‌ای (ORM)**

- سیستم‌های نگاشت شیء-رابطه‌ای (ORM) امکان می‌دهند:
  - تعیین نگاشت بین اشیاء زبان برنامه‌نویسی و تاپل‌های پایگاه داده
  - ایجاد خودکار تاپل‌های پایگاه داده هنگام ایجاد اشیاء
  - به‌روزرسانی/حذف خودکار تاپل‌های پایگاه داده هنگام به‌روزرسانی/حذف اشیاء
  - رابط برای بازیابی اشیاء با شرایط مشخص
  - تاپل‌ها در پایگاه داده پرس‌وجو می‌شوند و از آن‌ها اشیاء ساخته می‌شوند
- جزئیات در بخش ۹.۶.۲
- **Hibernate ORM** برای Java
- **Django ORM** برای Python

### توضیح آموزشی:
**ORM (Object-Relational Mapping)** محبوب‌ترین راه‌حل برای ادغام OOP و پایگاه داده است.

**مثال Django ORM:**
```python
# تعریف مدل (مثل CREATE TABLE)
class Student(models.Model):
    name = models.CharField(max_length=100)
    gpa = models.FloatField()
    courses = models.ManyToManyField('Course')

# استفاده - بدون هیچ SQL!
# ایجاد (INSERT INTO)
student = Student(name="علی", gpa=3.8)
student.save()

# پرس‌وجو (SELECT WHERE)
top_students = Student.objects.filter(gpa__gte=3.5)

# حذف (DELETE)
student.delete()
```

**مزایا:** سرعت توسعه، امنیت (جلوگیری از SQL Injection)، قابلیت تغییر پایگاه داده

**معایب:** عملکرد پایین‌تر برای پرس‌وجوهای پیچیده، «جادوی پنهان» که باگ‌یابی را سخت می‌کند

##  ۲۲ — داده‌های متنی


**داده‌های متنی**

- **بازیابی اطلاعات**: پرس‌وجوی داده‌های بدون ساختار
- مدل ساده پرس‌وجوی کلیدواژه‌ای: با توجه به کلیدواژه‌های پرس‌وجو، اسنادی که حاوی همه کلیدواژه‌ها هستند را بازیابی کن
- مدل‌های پیشرفته‌تر ارتباط اسناد را رتبه‌بندی می‌کنند
- امروزه پرس‌وجوهای کلیدواژه‌ای انواع مختلفی از اطلاعات را به‌عنوان پاسخ بازمی‌گردانند
  - مثلاً پرس‌وجوی "کریکت" معمولاً اطلاعاتی درباره مسابقات کریکت در حال برگزاری بازمی‌گرداند
- **رتبه‌بندی ارتباط**
  - ضروری است چون معمولاً اسناد زیادی با کلیدواژه‌ها مطابقت دارند

### توضیح آموزشی:
**بازیابی اطلاعات (IR)** علمی است که از دهه ۱۹۵۰ وجود داشته و امروزه پایه گوگل، Elasticsearch و موتورهای جستجو است.

**تکامل مدل‌های بازیابی:**
1. **مدل بولی (Boolean Model):** سند باید **همه** کلیدواژه‌ها را داشته باشد — نتیجه یا هست یا نیست
2. **مدل Vector Space (TF-IDF):** هر سند یک بردار عددی است، ارتباط با فاصله‌ی cosine محاسبه می‌شود
3. **مدل‌های احتمالاتی (BM25):** احتمال ارتباط سند با پرس‌وجو
4. **مدل‌های عصبی (امروز):** BERT، GPT — درک معنایی پرس‌وجو

**رتبه‌بندی چرا مهم است؟**
جستجوی "Python" در گوگل میلیون‌ها نتیجه دارد. بدون رتبه‌بندی، اولین نتیجه ممکن است درباره مار Python باشد نه زبان برنامه‌نویسی!

##  ۲۳ — رتبه‌بندی با TF-IDF


**رتبه‌بندی با TF-IDF**

- **عبارت (Term)**: کلیدواژه‌ای که در سند/پرس‌وجو رخ می‌دهد
- **فراوانی عبارت (TF)**: TF(d, t) — ارتباط عبارت t با سند d
  - یک تعریف: `TF(d, t) = log(1 + n(d,t)/n(d))`
  - n(d,t) = تعداد رخدادهای عبارت t در سند d
  - n(d) = تعداد کل عبارات در سند d
- **فراوانی معکوس سند (IDF)**: IDF(t)
  - یک تعریف: `IDF(t) = 1/n(t)`
- **ارتباط سند d با مجموعه عبارات Q:**
  - یک تعریف: `r(d, Q) = ∑ TF(d, t) × IDF(t)` برای هر t در Q
- تعاریف دیگر:
  - نزدیکی کلمات را در نظر می‌گیرند
  - کلمات توقف (Stop Words) اغلب نادیده گرفته می‌شوند

### توضیح آموزشی:
**TF-IDF** هوشمندانه‌ترین روش ساده برای اندازه‌گیری اهمیت یک کلمه در یک سند است.

**ایده اصلی:**
- کلمه‌ای که **در این سند زیاد** ظاهر می‌شود → مهم برای این سند (**TF بالا**)
- کلمه‌ای که **در همه اسناد** ظاهر می‌شود → کم‌اهمیت و عمومی (**IDF پایین**)

**مثال شهودی:**
- کلمه "the" در همه اسناد انگلیسی زیاد است → IDF نزدیک صفر → بی‌تأثیر
- کلمه "فتوسنتز" در تعداد کمی سند است → IDF بالا → مشخص‌کننده

**فرمول TF-IDF نهایی:**
```
TF-IDF(t, d) = TF(t, d) × IDF(t)
             = (تعداد کلمه در سند / کل کلمات سند) × log(کل اسناد / اسناد حاوی کلمه)
```

##  ۲۴ — رتبه‌بندی با TF-IDF (مثال عددی)


**رتبه‌بندی با TF-IDF**

- **پرس‌وجو Q:** The cat.
- **D1:** The cat is on the mat.
- **D2:** My dog and cat are the best.
- **D3:** The locals are playing.

**محاسبه TF:**
```
TF("the", D1) = 2/6 = 0.33
TF("the", D2) = 1/7 = 0.14
TF("the", D3) = 1/4 = 0.25

TF("cat", D1) = 1/6 = 0.17
TF("cat", D2) = 1/7 = 0.14
TF("cat", D3) = 0/4 = 0
```

### توضیح آموزشی:
**گام به گام محاسبه TF:**

برای D1 ("The cat is on the mat" — ۶ کلمه):
- "the": ۲ بار ظاهر شده → TF = 2/6 = 0.33
- "cat": ۱ بار ظاهر شده → TF = 1/6 = 0.17

برای D3 ("The locals are playing" — ۴ کلمه):
- "the": ۱ بار → TF = 1/4 = 0.25
- "cat": **اصلاً ظاهر نشده** → TF = 0/4 = 0

**نکته مهم:** در این تعریف ساده TF، کلمه توقف (Stop Word) مثل "the" که در همه جا هست همچنان TF بالایی دارد. اما IDF آن را خنثی می‌کند (مرحله بعد).

##  ۲۵ — رتبه‌بندی با TF-IDF (محاسبه IDF)


**رتبه‌بندی با TF-IDF**

- IDF با تقسیم تعداد کل اسناد بر تعداد اسنادی که حاوی یک کلمه هستند و گرفتن لگاریتم محاسبه می‌شود. اگر کلمه بسیار رایج باشد و در اسناد زیادی ظاهر شود، این عدد به صفر نزدیک می‌شود. در غیر این صورت به ۱ نزدیک می‌شود.

```
IDF(word) = log(تعداد کل اسناد / تعداد اسناد حاوی کلمه)
```

- **پرس‌وجو Q:** The cat.
- D1: The cat is on the mat. | D2: My dog and cat are the best. | D3: The locals are playing.

```
IDF("the") = log(3/3) = log(1) = 0
IDF("cat") = log(3/2) = 0.18
```

### توضیح آموزشی:
**IDF (Inverse Document Frequency)** اهمیت یک کلمه را در کل مجموعه اسناد اندازه می‌گیرد:

**تحلیل نتیجه:**
- `IDF("the") = log(3/3) = 0` ← "the" در همه ۳ سند هست! کاملاً بی‌معنا
- `IDF("cat") = log(3/2) ≈ 0.18` ← "cat" فقط در ۲ سند از ۳ هست، پس مقداری اهمیت دارد

**اگر کلمه نادر باشد (مثلاً فقط در ۱ سند از ۱۰۰۰):**
```
IDF = log(1000/1) = log(1000) = 3
```
→ IDF بالا = کلمه بسیار تخصصی و مهم

**چرا لگاریتم؟** لگاریتم از افزایش خطی جلوگیری می‌کند و مقیاس را متعادل نگه می‌دارد.

---

##  ۲۶ — رتبه‌بندی با TF-IDF (نتیجه نهایی)


**رتبه‌بندی با TF-IDF**

- ضرب TF و IDF امتیاز TF-IDF یک کلمه در یک سند را می‌دهد. هر چه امتیاز بالاتر باشد، آن کلمه در آن سند خاص ارتباط بیشتری دارد.

```
TF-IDF(word, document) = TF(word, document) × IDF(word)

TF-IDF("the", D1) = 0.33 × 0 = 0
TF-IDF("the", D2) = 0.14 × 0 = 0
TF-IDF("the", D3) = 0.25 × 0 = 0

TF-IDF("cat", D1) = 0.17 × 0.18 = 0.0306
TF-IDF("cat", D2) = 0.14 × 0.18 = 0.0252
TF-IDF("cat", D3) = 0    × 0 = 0

میانگین TF-IDF برای D1 = (0 + 0.0306) / 2 = 0.0153
میانگین TF-IDF برای D2 = (0 + 0.0252) / 2 = 0.0126
میانگین TF-IDF برای D3 = (0 + 0) / 2 = 0
```

### توضیح آموزشی:
**جمع‌بندی محاسبه برای پرس‌وجوی "The cat":**

| سند | TF-IDF("the") | TF-IDF("cat") | میانگین |
|-----|---------------|----------------|---------|
| D1 | 0 | 0.0306 | **0.0153** |
| D2 | 0 | 0.0252 | **0.0126** |
| D3 | 0 | 0 | **0** |

**رتبه‌بندی نهایی:** D1 > D2 > D3

**چرا D1 بهتر از D2؟** چون در D1 کلمه "cat" نسبت به کل سند بیشتر ظاهر شده (1 از 6 کلمه) در مقابل D2 (1 از 7 کلمه).

##  ۲۷ — رتبه‌بندی با TF-IDF (نتیجه‌گیری)


**رتبه‌بندی با TF-IDF**

- مشخص است که کلمه "the" به امتیاز TF-IDF هیچ سندی کمک نمی‌کند. این به این دلیل است که "the" در تمام اسناد ظاهر می‌شود و بنابراین یک کلمه بی‌ارتباط در نظر گرفته می‌شود.
- به‌عنوان نتیجه‌گیری، هنگام انجام پرس‌وجوی "The cat" روی مجموعه اسناد D1، D2 و D3، نتایج رتبه‌بندی‌شده به این ترتیب خواهند بود:
  - **D1:** The cat is on the mat.
  - **D2:** My dog and cat are the best.
  - **D3:** The locals are playing.
- توابع رتبه‌بندی با عملکرد بهتر در ادبیات وجود دارند، مثل Okapi BM25.

### توضیح آموزشی:
**Stop Words (کلمات توقف):**
کلمات بسیار رایج مثل "the", "is", "a", "an" معمولاً قبل از پردازش حذف می‌شوند چون:
- TF-IDF آن‌ها را به صفر نزدیک می‌کند
- حذف آن‌ها حجم ایندکس را کاهش می‌دهد
- جستجو را سریع‌تر می‌کند

**Okapi BM25 — جانشین پیشرفته‌تر TF-IDF:**
فرمول BM25 مشکلات TF-IDF را رفع می‌کند:
- TF را بهینه می‌کند (تکرار بیش از حد کلمه اثر کمتری دارد)
- طول سند را در نظر می‌گیرد (سند کوتاه‌تر با همان تعداد کلمه، مرتبط‌تر است)
- **Elasticsearch** از BM25 به‌عنوان الگوریتم پیش‌فرض استفاده می‌کند

##  ۲۸ — رتبه‌بندی با TF-IDF (کاربرد در یادگیری ماشین)


**رتبه‌بندی با TF-IDF**

- TF-IDF روشی برای مرتبط کردن هر کلمه در سند با عددی است که نشان می‌دهد هر کلمه در آن سند چقدر ارتباط دارد.
- این اعداد می‌توانند به‌عنوان ویژگی‌های مدل‌های یادگیری ماشین استفاده شوند.

### توضیح آموزشی:
**TF-IDF در یادگیری ماشین — بردار سند:**
هر سند به یک بردار عددی تبدیل می‌شود:

```
سند: "The cat sat on the mat"
↓ (پس از محاسبه TF-IDF برای هر کلمه)
بردار: [0, 0.17, 0.12, 0.09, 0, 0.15, ...]
         the  cat   sat   on   mat  ...
```

**کاربردها:**
- **طبقه‌بندی متن:** آیا این ایمیل اسپم است؟
- **خوشه‌بندی اسناد:** اسناد مشابه را کنار هم قرار بده
- **تشخیص زبان:** این متن به چه زبانی است؟
- **پیشنهاد محتوا:** اسناد مشابه به این مقاله را پیدا کن

**امروزه:** مدل‌های زبانی مثل BERT بازنمایی‌های بهتری (Word Embeddings) ارائه می‌دهند که معنا را هم در نظر می‌گیرند، اما TF-IDF هنوز برای کاربردهای سریع و ساده بسیار مفید است.

##  ۲۹ — رتبه‌بندی با استفاده از هایپرلینک


**رتبه‌بندی با استفاده از هایپرلینک**

- هایپرلینک‌ها سرنخ‌های بسیار مهمی درباره اهمیت ارائه می‌دهند
- گوگل PageRank را معرفی کرد، معیاری از محبوبیت/اهمیت بر اساس هایپرلینک‌های صفحات
  - صفحاتی که از صفحات زیادی لینک دریافت می‌کنند باید PageRank بالاتری داشته باشند
  - صفحاتی که از صفحات با PageRank بالاتر لینک می‌گیرند باید PageRank بالاتری داشته باشند
- با مدل پیاده‌روی تصادفی فرمول‌بندی شده:
  - فرض کن T[i,j] احتمالی باشد که یک پیاده‌روی تصادفی که در صفحه i است روی لینک به صفحه j کلیک می‌کند
  - با فرض برابری همه لینک‌ها: T[i,j] = 1/Ni
  - پس PageRank[j] برای هر صفحه j به‌صورت زیر تعریف می‌شود:
```
P[j] = δ/N + (1 - δ) × ∑ (T[i,j] × P[i])
```
  - که N = تعداد کل صفحات، و δ ثابتی معمولاً برابر ۰.۱۵

### توضیح آموزشی:
**PageRank** یکی از تأثیرگذارترین الگوریتم‌های تاریخ کامپیوتر است که توسط Larry Page و Sergey Brin (بنیان‌گذاران گوگل) در دانشگاه استنفورد طراحی شد.

**شهود پشت PageRank:**
- اگر یک سایت معتبر (مثل Wikipedia) به سایتی لینک بدهد، آن سایت بیشتر معتبر می‌شود
- اگر ۱۰۰۰ سایت ناشناخته به جایی لینک بدهند، ارزش کمتری دارد

**مدل پیاده‌روی تصادفی:**
تصور کنید یک کاربر بی‌هدف روی صفحات وب گشت می‌زند:
- با احتمال `(1-δ)`: روی یک لینک تصادفی کلیک می‌کند
- با احتمال `δ`: به یک صفحه کاملاً تصادفی می‌پرد (جلوگیری از گیرافتادن)

PageRank صفحه j = احتمال اینکه این کاربر در صفحه j باشد

**δ = 0.15 چیست؟** یعنی ۱۵٪ اوقات کاربر به جای کلیک روی لینک، یک URL تصادفی وارد می‌کند.

##  ۳۰ — رتبه‌بندی با هایپرلینک (ادامه)


**رتبه‌بندی با استفاده از هایپرلینک (ادامه)**

- تعریف PageRank دوری (circular) است، اما می‌توان آن را به‌صورت مجموعه‌ای از معادلات خطی حل کرد
- روش تکراری ساده خوب عمل می‌کند:
  1. همه P[i] = 1/N مقداردهی اولیه کنید
  2. در هر تکرار از معادله `P[j] = δ/N + (1-δ) × ∑(T[i,j] × P[i])` برای به‌روزرسانی P استفاده کنید
  3. زمانی که تغییرات کوچک شد یا به حدی رسید (مثلاً ۳۰ تکرار) متوقف شوید
- معیارهای دیگر ارتباط نیز مهم هستند. برای مثال:
  - کلیدواژه‌ها در متن لنگر (anchor text)
  - تعداد دفعاتی که کاربران پس از جستجوی یک پرس‌وجو روی یک لینک کلیک می‌کنند

### توضیح آموزشی:
**حل تکراری PageRank:**

```python
# شبه‌کد PageRank
N = تعداد صفحات
P = [1/N] * N  # مقداردهی اولیه

for iteration in range(30):
    P_new = []
    for j in range(N):
        # جمع PageRank همه صفحاتی که به j لینک می‌دهند
        backlink_sum = sum(T[i][j] * P[i] for i in range(N))
        P_new[j] = 0.15/N + 0.85 * backlink_sum
    P = P_new
```

**چرا ۳۰ تکرار کافی است؟**
در عمل، PageRank به مقدار ثابتی همگرا می‌شود. تغییرات بین تکرار ۲۵ و ۳۰ معمولاً کمتر از ۰.۰۰۰۱ است.

**Anchor Text (متن لنگر):**
اگر ۱۰۰۰ سایت با متن "بهترین رستوران تهران" به سایتی لینک بدهند، گوگل این صفحه را برای این عبارت بسیار ارتباط‌دار می‌داند — حتی اگر خود صفحه این کلمات را نداشته باشد!

##  ۳۱ — رتبه‌بندی با هایپرلینک (تفسیر)


**رتبه‌بندی با استفاده از هایپرلینک**

- این فرمول PageRank یک صفحه را با جمع درصدی از مقدار PageRank تمام صفحاتی که به آن لینک می‌دهند محاسبه می‌کند. بنابراین بک‌لینک از صفحات با PageRank بیشتر ارزش بیشتری دارند. علاوه بر این، صفحاتی با لینک‌های خروجی بیشتر، بخش کوچک‌تری از PageRank خود را به هر صفحه وب لینک‌شده منتقل می‌کنند.
- طبق این فرمول، سه عامل اصلی که PageRank یک صفحه را تأثیر می‌دهند عبارتند از:
  - تعداد صفحاتی که به آن بک‌لینک می‌دهند
  - PageRank صفحاتی که به آن بک‌لینک می‌دهند
  - تعداد لینک‌های خروجی در هر یک از صفحاتی که به آن بک‌لینک می‌دهند

### توضیح آموزشی:
**مثال عددی PageRank:**
تصور کنید:
- صفحه A با PageRank = 10 به صفحه C لینک دارد (تنها لینک)
- صفحه B با PageRank = 1 به صفحه C لینک دارد (یکی از ۱۰۰ لینک B)

کمک A به C: `10 × (1/1) = 10`
کمک B به C: `1 × (1/100) = 0.01`

بنابراین لینک از یک صفحه معتبر با لینک‌های کم خروجی ارزشمندتر از هزاران لینک از صفحات ضعیف است.

**کاربرد در SEO (بهینه‌سازی موتور جستجو):**
این درک به صنعت SEO کمک کرد که روی کیفیت بک‌لینک‌ها تمرکز کند نه کمیت.

##  ۳۳ — اثربخشی بازیابی


**اثربخشی بازیابی**

- **معیارهای اثربخشی:**
  - **دقت (Precision)**: چه درصدی از نتایج برگشتی واقعاً مرتبط هستند؟
  - **فراخوانی (Recall)**: چه درصدی از نتایج مرتبط بازیابی شدند؟
  - در تعداد مشخصی از پاسخ‌ها، مثلاً precision@10، recall@10
- **پرس‌وجوی کلیدواژه‌ای روی داده‌های ساختاریافته و پایگاه دانش:**
  - مفید است اگر کاربران طرح‌واره را نمی‌دانند، یا طرح‌واره از پیش تعریف نشده است
  - می‌توان داده را به‌صورت گراف نمایش داد
  - کلیدواژه‌ها با تاپل‌ها تطابق پیدا می‌کنند
  - جستجوی کلیدواژه تاپل‌هایی که کلیدواژه‌ها را دارند و به هم نزدیک هستند بازمی‌گرداند
  - مثال: در پایگاه داده دانشگاه با پرس‌وجوی "Zhang Katz"، Zhang با یک دانشجو و Katz با یک استاد مطابقت پیدا کرده و رابطه استاد راهنما آن‌ها را به هم وصل می‌کند

### توضیح آموزشی:
**دقت و فراخوانی — معیارهای کلاسیک IR:**

فرض کنید ۱۰۰۰ سند داریم، ۴۰ تای آن‌ها مرتبط با پرس‌وجو هستند، و سیستم ۵۰ سند برمی‌گرداند که ۲۰ تایش واقعاً مرتبط هستند:

```
Precision = 20/50 = 0.40 (40% از نتایج برگشتی مرتبط بودند)
Recall    = 20/40 = 0.50 (50% از نتایج مرتبط را پیدا کردیم)
```

**تنش بین دقت و فراخوانی:**
- اگر **همه** اسناد را برگردانیم: Recall = 1.0، Precision = 0.04
- اگر **فقط مطمئن‌ترین** را برگردانیم: Precision بالا، Recall پایین

**precision@10** یعنی: از ۱۰ نتیجه اول، چند تا واقعاً مرتبط بودند؟ این معیار برای موتورهای جستجو مهم‌ترین است چون کاربران معمولاً فقط صفحه اول نتایج را می‌بینند.

**F1 Score:** میانگین هارمونیک دقت و فراخوانی، معیاری که هر دو را در نظر می‌گیرد:
```
F1 = 2 × (Precision × Recall) / (Precision + Recall)
```

*پایان فصل ۸ — انواع داده پیچیده*
